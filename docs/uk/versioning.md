# 🛡️ Стратегія Версіонування NanWeb

Цей документ описує правила управління версіями в монорепозиторії NanWeb <v name="version">v3.0.0</v> (<v name="release_name">Sovereign Audit</v>). Ми використовуємо гібридний підхід: синхронізація екосистеми при великих релізах та атомарні оновлення для пакетів.

## 🏛️ 1. Глобальна Версія (Ecosystem)

Весь монорепозиторій орієнтується на **Main Release Tag** (зараз <v name="version">v3.0.0</v>). Джерелом істини для цього значення є файл `docs/_.md`.

- **Синхронізація**: При переході на нову мажорну версію (v3), всі стабільні пакети та додатки отримують цю версію у своїх `package.json`.
- **Ієрархія Оновлень**: 
  - **Локальний патч** (напр. `3.0.1` в одному пакеті) НЕ потребує негайної зміни глобальної версії в `docs/_.md`.
  - **Глобальний Sync** варто робити при випуску нового функціоналу або значних змінах у декількох модулях. Глобальна версія слугує "найменшим спільним знаменником" стабільності.

## 📦 2. Локальні Версії та Залежності

Кожен пакет має власну версію, але завдяки протоколу **`workspace:*`**, будь-яке оновлення автоматично інтегрується в екосистему.

### Сценарій: Перехід пакету в нову Екосистему
Якщо пакет `@nan0web/ai` має версію `1.5.0`, а монорепо переходить на `v3.0.0`:
1. Наступна версія AI стає **3.0.0** (або **3.0.1** після першого патчу).
2. Ми робимо стрибок версії, щоб підтвердити: цей пакет тепер повністю відповідає стандартам "Sovereign Audit".
3. Більше ніяких `1.x` релізів для цього пакету не створюється.

## 🏗️ 3. Sovereign Inheritance (Синхронізація Документації)

Ми використовуємо двигун `SyncDocsApp` для автоматичної проливки версій у документацію.

- **Джерело**: `docs/_.md` (поле `version`).
- **Ціль**: Усі `.md` файли в папці `docs/`.
- **Механізм**: Тег `<v name="version">v3.0.0</v>` автоматично замінюється на значення з джерела.
- **Команда**: `node bin/sync-docs.js`.

## 🚀 4. Процес Релізу (AGRP Workflow)

Релізи розділяються на два рівні за своїм впливом:

### Рівень Екосистеми (Global)
- **Шлях**: `releases/{major}/{minor}/vX.Y.Z/` у корені монорепо.
- **Вплив**: Архітектурні зміни, загальна документація, крос-пакетні фічі.

### Рівень Пакету (Local)
- **Шлях**: `packages/{name}/releases/{major}/{minor}/vX.Y.Z/`.
- **Вплив**: Фічі та багфікси конкретного модуля.

### Етапи Релізу:
1. **Планування**: Створення `task.md` (Specification) та `task.spec.js` (Contract).
2. **Аналіз**: `node bin/audit.js` перевіряє архітектурну цілісність.
3. **Bump**: 
   - Використовуйте `bin/bump.js` для оновлення версії пакету.
   - Встановіть версію в `docs/_.md` та запустіть `node bin/sync-docs.js`.
4. **Закриття**: Після схвалення Архітектором, `task.spec.js` мігрує в `src/test/releases/...` як **регресійний тест**.
5. **Публікація**: `pnpm publish`. Версія в NPM має збігатися з `package.json`.

## 🏷️ 5. Формат Тегів у Git

Оскільки всі пакети знаходяться в одному репозиторії, ми використовуємо префікси для забезпечення унікальності тегів:

### Глобальні Теги (Ecosystem)
- **Формат**: `vX.Y.Z` (наприклад, `v3.0.0`).
- **Застосування**: Позначає стан всього монорепозиторію. Збігається з версією в `docs/_.md`.

### Локальні Теги (Packages)
- **Формат**: `{package-name}@vX.Y.Z` (наприклад, `@nan0web/ai@v3.0.1`).
- **Застосування**: Позначає реліз конкретного пакету. Збігається з версією в `package.json` модуля.
