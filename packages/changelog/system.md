# 📦 Проєкт `@nan0web/changelog`

## 🎯 Мета

Надавати простий, типізований та протестований API для роботи з *CHANGELOG.md*:

- парсинг існуючих записів
- отримання версій, змін та дат
- додавання нових версій і змін
- генерація markdown‑подання для документування

## 🤖 Інструкція для LLM

1. **Контекст** – файл `system.md` є єдиним джерелом правил для LLM.  
   Працюйте лише в межах описаних функцій (`Changelog`, `Version`, `Change`), їх типів і тестів.
2. **Генерація коду** – дотримуйтеся **vanilla JavaScript** з JSDoc‑тегами, без `;` в кінці рядків.  
   Якщо щось незрозуміло, відповідайте `Контекст загублено. Я не спроможній на це. Обери більшу модель.`
3. **Помилки** – будь‑яка невідповідність формату відповідайте `NO PROTOCOL RULE` і уточнюйте, що порушено.
4. **Валідація** – усі зміни мають проходити проєктні тести (`npm test`). Якщо тест падає, повертайте повний вивід тесту.
5. **Висновок** – після успішного виконання задач додавайте `#.` (дві нові строки, `#`, крапка, новий рядок).

## 👩‍💻 Правила розробки

| Правило               | Опис                                                                                                                  |
| --------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **Файлова структура** | `src/` – вихідний код, `src/*.test.js` – юніт‑тести, `types/` – декларації `.d.ts`, `system.md` – системний протокол. |
| **Імпорти**           | Спочатку `node:`‑модулі, потім `@nan0web/*`, потім локальні.                                                          |
| **JSDoc**             | Описуйте всі експортовані функції/класи (`@param`, `@returns`, `@type`).                                              |
| **Тестування**        | Кожна функція/метод – мінімум один `it`‑тест. Пишіть позитивні та негативні сценарії у циклі.                         |
| **Команди NPM**       | `npm test` – усі тестові файли, `npm run build` – генерує декларації, `npm run play` – демонстрація API.              |
| **Версії**            | Використовуйте `Version`‑клас для парсингу/генерації SemVer. Формат: `## [x.y.z] - YYYY‑MM‑DD`.                       |
| **Додавання змін**    | `Changelog.addChange(change)` автоматично створює/оновлює версію, розділи та списки.                                  |
| **README**            | `src/README.md.js` – документ, який одночасно тестує і генерує `README.md`.                                           |
| **Лінт**              | Використовуйте `node --check` під час CI, лінт не допускає `;` в кінці рядка.                                         |

## ⚙️ Приклад використання (в `play/main.js`)

```js
import Changelog from '@nan0web/changelog/src/Changelog.js'
import Change from '@nan0web/changelog/src/Change.js'

const log = new Changelog()
log.init() // створює базовий шаблон
log.parse(SAMPLE) // парсить існуючий changelog
const newChange = Change.from({
  added: '- Новий функціонал',
  major: 1,
  minor: 4,
  patch: 0,
  date: new Date(),
})
log.addChange(newChange) // додає нову версію та зміну
console.log(log.document.toString())
```

## 📦 Публікація

```bash
pnpm run build          # генерує типи в ./types
npm publish             # після успішних тестів і `npm run build`
```

## 🛠️ Додаткові інструменти

- `@nan0web/log` – кольорове логування.
- `@nan0web/markdown` – парсер/генератор markdown‑елементів.
- `@nan0web/test` – утиліта для запуску та парсингу тестових результатів.

## ✅ Критерії успішності

- Усі тести (`npm test`) проходять без помилок.
- `npm run build` успішно створює типи.
- `npm run play` виводить коректний markdown‑файл.
- `README.md` генерується з `src/README.md.js` і відповідає документації.
