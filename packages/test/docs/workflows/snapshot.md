# 📸 Ворклоу: Snapshot Тестування

Цей workflow використовується виключно для авто-запуску, оновлення та верифікації вже створених Snapshot-тестів (Golden Master).

> **Зв'язок з іншими workflow:**
>
> - Для **створення** нової пісочниці та інфраструктури тестів використовуйте: `sandbox-template.md`
> - Для **ручної розробки та візуальної перевірки** компонента: `sandbox-verify.md`
> - Цей workflow (`snapshot.md`) потрібен для масової перевірки існуючих тестів або оновлення еталонів після запланованої зміни дизайну всієї системи.

Всі зміни виводу мають бути свідомо підтверджені.

## 1. Sovereign Visual Orientir (100% Істина)

Снепшот — це "Golden State", який визначає правильність.

- **Broken State Principle**: Будь-яке відхилення (навіть на 1 пробіл) ламає білд.
- **Візуальний доказ**: Ніколи не припускай, що код працює. Доведи це сніпшотом.

## 2. Типи Snapshot-тестів

- **CLI-графіка (Термінал)**: `snapshots/play/*.snap`. Використовує утиліти `@nan0web/play` для нормалізації виводу (`normalizeOutput()`, ANSI strip, `[SPINNER_FRAME]`, тощо).
- **Web UI E2E (Playwright)**: Скріншоти браузера `snapshots/play/web/*.png` через `verifyScreenshot()`.

## 3. Перевірка Snapshot (Verify)

Запуск верифікації без зміни файлів:

// turbo

```bash
npm run test:snapshot
```

## 4. Оновлення Snapshot (Golden Master Update)

Якщо зміни у UI/CLI **заплановані та коректні**, необхідно оновити базові сніпшоти. Цей скрипт оновить всі зламані сніпшоти еталонним новим виводом:

// turbo

```bash
npm run test:snapshot:update
```

Або, якщо такого скрипта немає у `package.json`:

// turbo

```bash
UPDATE_SNAPSHOTS=1 node --test play/snapshot.test.js
```

## 5. Генерація Галереї (Component Sandbox)

Для візуального огляду UI-компонентів:

// turbo

```bash
npm run gallery
```

## 6. Чеклист Змін

1. Переглянь через `git status`, які `.snap` або `.png` файли змінилися.
2. Зазирни у `git diff`, щоб переконатись, що зміна є очікуваною (наприклад, змінився текст, а не "зламалася" верстка дерева чи таблиці).
3. Переконайся, що `npm run test:all` (якщо є) також успішний.

## 7. Аудит Снепшотів (Snapshot Audit)

Якщо користувач викликає повідомлення з проханням проаудіювати снепшоти (або `/snapshot-audit`), ти (агент) мусиш суворо виконати наступні кроки, не пропускаючи жодного файлу:

### Крок 1: Збір існуючих снепшотів

1. Використай інструменти читання директорій для збору всіх файлів еталонів. У рамках пакета вони зазвичай знаходяться у `snapshots/` (наприклад: `snapshots/play/` та `snapshots/sandbox/`) або у вкладених теках.
2. Відсортуй їх за мовами (`en`, `uk`) та типами, щоб аудит був систематичним.

### Крок 2: Збір очікуваних результатів (Контракт)

1. По ідеї для кожного снепшота у нас має бути розуміння, як воно має працювати. Це можуть бути дані `data/**/*.{yaml|nan0|json|md}`, або документація компонента у релізі (`releases/`), файлі `project.md`, `system.md` чи `src/README.md.js`. Ознайомся з очікуваною поведінкою.
2. Зрозумій "Очікуваний результат" (Expected Result) для кожного специфічного екрану.

### Крок 3: Покроковий аналіз снепшотів

Для **КОЖНОГО** знайденого файлу `.snap` або `.webp`:

1. Прочитай вміст файлу (для текстових .snap) за допомогою інструменту `view_file` або переглянь візуально.
2. **Перевір на помилки виконання (Red Flags):** Шукай у тексті снепшоту такі рядки:
   - `TypeError`, `Error`, `ReferenceError`, `[object Object]`, `NaN`, `undefined`, сирі ANSI-коди (`\x1b`).
3. **Перевір на семантику та i18n:** Якщо сніпшот має `.uk.snap`, переконайся, що текст ДІЙСНО перекладено українською (відсутність англійських лейблів).
4. **Перевір на відповідність очікуванням:** Чи виглядає цей вивід як завершений і правильний UI? Якщо це "Tree", чи дійсно відображається дерево і всі іконки на своїх місцях?
5. Зроби короткий висновок у своєму `thought` процесі.

### Крок 4: Формування Звіту

Після того, як **всі** файли проаналізовані, згенеруй для користувача **Markdown-звіт у вигляді Артефакту** з такими розділами:

- 🟢 **Чисті Снепшоти** (короткий підсумок, скільки файлів ідеальні).
- 🔴 **Снепшоти з Помилками** (список файлів із прямою цитатою помилки).
- 🟡 **Підозрілі випадки** (випадки де візуальний вивід виглядає зрізаним).

> **Критично:** Це перевірка на уважність. Часто `[object Object]` ховаються глибоко. Ніколи не вигадуй вивід. Якщо компонент не задокументовано чітко, оцінюй його з точки зору здорового глузду та загальної узгодженості (consistency) NaN•Web UI.
