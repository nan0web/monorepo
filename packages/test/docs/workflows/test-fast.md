# Протокол швидкого тестування (Shortest Path)

Цей воркфлоу визначає найкоротший та найшвидший шлях для тестування під час розробки. Штучний інтелект (Агенти/LLM) **зобов'язані використовувати найбільш специфічний і швидкий скрипт** замість запуску всього повільного конвеєра (`test:all`), щоб економити час та ресурси в ході вирішення задач.

## 🚀 Конкретні команди (Shortest Path)

Обирай скрипт залежно від того, що саме ти щойно змінив:

1. **`npm run test:unit` (Сценарне тестування OLMUI)**
   - **Що робить:** Швидко запускає `test:unit` (Сценарні та Unit/Contract тести в `src/**/*.test.js`).
   - **Коли використовувати:** Змінено бізнес-логіку додатків чи моделей (`src/`).
   - 🧠 **Сценарне тестування в OLMUI:** Оскільки кожна Модель (Model-as-Schema) інкапсулює **повний життєвий цикл і логіку цілого додатка** (через генератор `run()`), ми виконуємо *сценарне тестування* саме на рівні Unit-тестів. Замість повільного проклікування кнопок у браузері, ми проходимо всі етапи роботи додатка (всі `yield` інтенції) в ізольованому `node --test`. Це перевіряє всю поведінку та сценарії використання за долі мілісекунди (без Playwright чи реального UI).

2. **`npm run test:ssg`**
   - **Що робить:** Запускає `test:ssg` (перевірка згенерованих статичних HTML документів та YAML).
   - **Коли використовувати:** Коли ти додав новий `.yaml` компонент, змінив маппінг категорій для генерування документації чи змінив логіку в `generate-pages.js`.

3. **`npm run build`** (саме `build`, не `test:build`)
   - **Що робить:** Перевіряє TypeScript (запускає `tsc` перевірку для гарантії, що всі типи та JSDoc вірні).
   - **Коли використовувати:** Як тільки ти змінив чи оновив типи JSDoc (`/** @param ... */`), або змінив аргументи, що впливають на типізацію.

4. **`npm run test:e2e`**
   - **Що робить:** Повільні Playwright тести в браузері.
   - **Коли використовувати:** Тільки коли змінено реальний DOM, інтерактивність UI компонентів чи CSS. Логіка додатку перевіряється в `test:unit`, а тут лише *візуальний/браузерний рендеринг*. Запускається ЯК ОКРЕМИЙ ТЕСТ тільки після успіху `test:unit`.

## 🛡️ Стандартні комбіновані скрипти (Commit / Release)

Ці скрипти об'єднують перевірки на різних етапах готовності:

1. **`npm test` (або `npm run test`)**
   - Це стандартна базова перевірка, яка визначає чи пройде **git commit** (через husky hook).
   - Включає в себе: `test:unit`, `test:ssg` та `build`. 
   - Використовуй тоді, коли хочеш перевірити, чи все готово до наступного коміту.

2. **`npm run test:all`**
   - Об'єднує **всі абсолютно** тести, включаючи повільні: `test`, `test:docs`, `test:play`, `npm run test:e2e` та `npm run knip`.
   - Запускається **виключно перед генеральним релізом (`/npm-release`) або публікацією**, або в кінці масштабного завдання. Під час ітеративної розробки не використовуй `test:all`, оскільки він займає багато часу!
