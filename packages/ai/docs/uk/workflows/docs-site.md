# 📚 Ворклоу: Генератор Документації (Docs-Site)

Цей workflow призначений для створення інтерактивного сайту документації будь-якого пакету екосистеми.
Він використовує `nan0web.app` (внутрішній рушій замість Vitepress), забезпечуючи миттєвий зворотний зв'язок, інтерактивні приклади (Playground/CLI) та 100% сумісність із нашою філософією Data-Driven архітектури.

## 1. Концепція "Живої Документації" (Living Docs)

За документацію більше не потрібно "доглядати" вручну. Вона генерується автоматично, витягуючи сутність пакету з джерел істини:

- **Архітектура (`project.md`):** Опис, цілі, 5 фаз. Структуровано як роутер у корні, з перекладами у `docs/uk/project.md` та `docs/en/project.md`.
- **Презентація (`README.md`):** Встановлення, швидкий старт.
  - Кореневий `README.md` залишається англійським оригіналом для NPM і GitHub (з додаванням лінку `> 🇺🇦 [Читати українською](./docs/uk/README.md)` зверху).
  - **КРИТИЧНЕ ПРАВИЛО ЛОКАЛІЗАЦІЇ:** Агент **ОБОВ'ЯЗКОВО** створює точну копію (клон) кореневого `README.md` у файлі `docs/en/README.md`.
  - Це дозволяє рушію `nan0web.app` дуже просто і лінійно рендерити виключно папку `docs/` та перемикати мови (`docs/en/` та `docs/uk/`), не створюючи складних алгоритмів пошуку кореневих файлів.
  - Переклад розміщується у `docs/uk/README.md`.
- **API та Типи:** `types/*.d.ts` або JSDoc у `src/` (Публічний інтерфейс).
- **Інтерактив:** Папка `play/` або `playground/` (Працюючі демо, CLI/Web).

## 2. Атрибутика Контексту для Агента

- **Вимога до чату:** Звичайний або 🔴 NEW CHAT.
- **Необхідні інструменти:**
  - `apps/nan0web.app` для оркестрації та керування станом відображення.
  - Попередня індексація пакета (`bin/indexator.js`).

## 3. Завдання Агента (Architechnomag)

При виклику `@[/docs-site]`, агент зобов'язаний:

1. **Сканування Пакету:**
   - Зчитати усі наявні Markdown файли.
   - Зчитати розкреслений функціонал із `.d.ts` файлів (типи повністю описують API, тому не треба дивитись у `src/`).
   - Визначити наявність папки `play/` (для прикладу CLI/Web).

2. **Використання nan0web.app як Рушія (Runner):**
   - На відміну від старих систем (як VitePress), `nan0web.app` самостійно клонує кореневі файли в пам'ять або тимчасову базу і самостійно збирає структуру сторінок під час запуску.
   - Агенту потрібно лише додати NPM-скрипти до `package.json` пакету:
     ```json
     {
       "scripts": {
         "docs:dev": "npx nan0web.app serve --dsn docs/ --port 8080",
         "docs:build": "npx nan0web.app build --dsn docs/"
       }
     }
     ```
   - За замовчуванням `nan0web.app` шукає теку `docs/`, клонує `README.md` з кореня, і об'єднує все це у спільну віртуальну файлову систему.

3. **Синхронізація (Автоматичний білд):**
   - Переконатися, що у пакеті налаштована команда `npm run docs` або `npm run docs:dev`.
   - Це має запускати `nan0web.app` зі вказівкою кореня `docs`: `npx nan0web.app serve --root docs/`.

## 4. Вимоги до Інтерфейсу `nan0web.app` Docs

- **Блискавична швидкість:** Відсутність процесів компіляції Markdown (Zero-build / Runtime render).
- **Темна/Світла Тема:** Нативна підтримка брендових кольорів NaN.
- **Playground (Пісочниця):** Усі приклади коду мають кнопку "Run", щоб виконатися локально або у браузерному середовищі (Sandboxed) без перезавантаження.

## 5. Результат і Рапорт

1. Перевірено та структуровано документацію пакету.
2. Створено або оновлено `docs.yaml` чи структуру `.md` файлів, готову для згодовування рушію.
3. Сповіщення користувачу:
   > ✅ Структура живої документації створена та інтегрована з рушієм `nan0web.app`. Запусти `npm run docs` у пакеті, щоб побачити магію.
