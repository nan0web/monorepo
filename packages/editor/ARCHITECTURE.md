# Editor Architecture: Sovereign Data CMS

Цей документ описує архітектуру універсального редактора (`editor.app` та `@nan0web/editor`), який забезпечує керування даними платформи `nan•web` у мультиплатформеному середовищі.

## 1. Філософія: Model-Driven UI

Редактор не знає заздалегідь про структуру даних. Він будує інтерфейс динамічно, аналізуючи **Доменні Моделі**.

- **Schema Interpretation**: Кожна модель (`Model.schema`) містить метадані про поля (типи, валідація, допомога, плейсхолдери).
- **Data Composition**: Використання `Data.merge()` та системних операторів:
  - **Inheritance**: Автоматичне успадкування даних від батьківських директорій.
  - **Reference ($ref)**: Атомні посилання на інші об'єкти для динамічної збірки.
  - **Clear ($clear)**: Оператор для повного очищення масивів або об'єктів при мерджі.
    (Згідно зі стандартом `@nan0web/db@1.3.0`).

## 2. Recursive Modals (Stack-Based Navigation)

Для редагування складних, деревоподібних структур використовується концепція **"Вікно у вікні"**:

- **Stack of Modals**: Кожне нове редагування відкриває модальний шар над попереднім.
- **Maximum Depth**: Редактор ПОВИНЕН обмежувати глибину рекурсії (Default: 7 рівнів) для збереження продуктивності та юзабіліті.
- **Context Return**: Закриття верхнього шару повертає оновлений об'єкт або його `$ref` у батьківське поле.

## 3. Catalog & Reference Management

Робота з ідентифікаторами спрямована на максимальну швидкість вибору:

- **Autocomplete**: Вибір об'єкта зі списку (catalog) реалізується через розширений `Autocomplete` компонент.
- **ToString Serialization**:
  - Кожна модель ПОВИННА реалізовувати метод `toString()` (або його аналог у схемі).
  - Для CLI: Використання табуляції (`ID \t Title`) дозволяє адаптеру автоматично форматувати випадаючий список у колонки.
- **Dynamic Extensibility**: Компонент вибору в каталозі може бути розширений для відображення додаткових полів (status, date, preview).

## 3. Multi-Platform Adapters

Редактор реалізований як **Core Logic** (в `packages/editor/src/core`), який розширюється адаптерами:

| Платформа   | Технологія        | Механізм відображення                                               |
| :---------- | :---------------- | :------------------------------------------------------------------ |
| **Web**     | Lit / React       | Динамічні форми на базі Web Components.                             |
| **VS Code** | Webview           | Інтеграція через `EditorBridge.js` та Custom Text Editor API.       |
| **CLI**     | `@nan0web/ui-cli` | Інтерактивні форми в терміналі з підтримкою `TAB` та пошуку.        |
| **Mobile**  | SwiftUI / Compose | Нативні адаптери, що рендерять JSON-опис полів у системні UI-блоки. |
| **Bots**    | Telegram App      | Inline-кнопки та діалоги для швидкого редагування (mini-apps).      |

## 4. Visual Review & Verification (Blueprint & Sticker Protocol)

Ми використовуємо метафору архітектурного креслення для стабільності інтерфейсу:

- **Blueprints (Креслення)**: `npm run test:snapshot` створює еталонні знімки екрана. Вони сприймаються як технічне завдання, яке "надруковане" у великому форматі для перевірки.
- **Stickers (Стікери)**:
  - Кожен затверджений стан отримує віртуальний стікер `$ok`.
  - Стікери наклеюються зверху на "креслення" (скріншот) у `PLAY_GALLERY.md`.
  - Тільки компоненти зі стікерами `$ok` можуть бути інтегровані в основну гілку.
- **Interactive Review**: Розширення `vscode-nan0` дозволяє "клеїти" стікери прямо в редакторі через кнопки `[Approve]`. Це створює інтерактивний цикл TDD: зміна коду -> нове креслення -> перевірка -> стікер.

## 5. Synchronization (Offline-First)

Використання `DBBrowser` з шаром `IndexedDB` дозволяє:

- Редагувати дані без інтернету.
- Ставити зміни у чергу (Unsynced queue).
- Автоматично надсилати `POST/PUT/DELETE` запити при відновленні мережі.
