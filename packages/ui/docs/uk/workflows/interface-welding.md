# 🛠 Контракт Зварювання (Interface Welding)

Цей workflow формалізує правило безперервної верифікації графічних інтерфейсів (Lit, React, CLI-меню тощо) як захист від "галюцинацій" Агента та зламаного UI.

## 1. Суть Контракту

Будь-яка візуальна розробка (UI) в екосистемі `nan.web` вважається дефектною (браком), якщо вона не має миттєвої, паралельно написаної системи зворотного зв'язку (тесту).
**Агенту СУВОРО ЗАБОРОНЕНО генерувати або модифікувати будь-який UI-компонент, якщо в тому ж кроці (чи коміті) не генерується тест, який його перевіряє.**

## 2. Як це працює для різних платформ

### Для Web-компонентів (React / Lit / Vanilla UI):

Як тільки Агент створює `FancyButton.jsx` або `SomeWidget.js`, він **зобов'язаний** в тій самій ітерації:

1. Згенерувати або оновити відповідний Playwright/Snapshot тест (наприклад, `play/snapshot.test.js` з інстансом цього компонента).
2. Або інтегрувати компонент у веб-пісочницю пакета (`play/app.js`, `play/index.html`) і записати у `play.e2e.js` команду на зняття снепшоту/перевірку відображення.
3. Якщо компонент працює з даними, Агент зобов'язаний згенерувати для нього `mockData` або оновити конфіги в `db-fs` (`data/index.yaml`).

### Для CLI-меню (`@nan0web/ui-cli`):

Якщо додається нова команда в термінал (`adapter.select` або новий екран):

1. **Обов'язково** паралельно додається сценарій його виклику до `play/snapshot.test.js` та `play/main.js` (у `DEMO_MAP`).
2. Код перевіряється запуском `npm run test:snapshot:update`.

## 3. Правило "Dumb Interfaces" (Дзеркала Даних)

Контракт зварювання гарантує, що UI є "тупим" (Dumb View).

- Усі переклади беруться тільки з бази через функції локалізації (заборонено тернарні JS-переклади `lang === 'uk' ? 'Так' : 'Yes'`).
- UI-компоненти лише "віддзеркалюють" вміст бази даних (YAML/JSON конфіги).
- Якщо логіку можна перенести в бізнес-модель (`Model-as-Schema`) — її **ОБОВ'ЯЗКОВО** треба перенести туди, залишивши React/Lit/CLI компонент чистим.

## 4. Патерн розробки для Агента

Коли Користувач просить: _"Зроби компонент картки банку"_
Агент робить:

1. Створює/Змінює файл компонента (`BankCard.jsx`).
2. Додає цей компонент у `play/app.js` (або `play/main.js` для CLI).
3. Пише фейкові дані в `play/data/uk/index.yaml` (якщо потрібно).
4. Оновлює Snapshot-тести або E2E, щоб він зробив "скріншот" нового компонента.
5. Рапортує про успіх.
