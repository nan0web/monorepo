# Специфікація: Кешування в пам'яті для DBFS (Memory Cache)

## Мета
Значно пришвидшити роботу `DBFS.fetch()` та `sync-data.js`, зменшивши кількість дорогих системних викликів (I/O операцій `fs.stat`, `fs.readFile`, рекурсивного пошуку `_globals.yaml` та `_.yaml`). Впровадити in-memory кеш для зчитування даних та метаданих, що дозволить миттєво обробляти тисячі файлів під час ініціалізації додатків.

## Проблема
Поточна реалізація `DB.js` та `DBFS.js` при кожному виклику `fetch(uri)`:
1. Викликає `statDocument` для самого файлу.
2. Викликає `statDocument` для можливих розширень (yaml, json, nan0), щоб розв'язати індекси.
3. Рекурсивно піднімається вгору по дереву директорій, перевіряючи наявність `_.yaml` (наслідування) та `_globals.yaml` через множинні виклики `fs.exists` та `fs.stat`.
Ці системні виклики множаться на кількість файлів, через що обробка 60 карток займає ~600+ мс замість 20 мс (чистий час парсингу).

## Архітектурне Рішення (План дій)

### 1. Додавання шару кешування в `DB.js` / `DBFS.js`
- Впровадити `Map` для збереження результатів `statDocument` (метадані) та `loadDocument` (чисті розпарсені дані).
- Додати опцію `useCache: boolean` в параметри ініціалізації `DBFS` (за замовчуванням `true` для production та `sync-data`, або з можливістю включення через `cache.enable()`).

### 2. Кешування `statDocument` (Найбільше I/O навантаження)
- При виклику `statDocument(uri)`, перевіряти наявність `uri` у `this._statCache`.
- Якщо є — повертати кешований об'єкт `DocumentStat`.
- Якщо ні — робити системний виклик `FS.stat`, створювати об'єкт і зберігати в кеш.

### 3. Кешування розпарсених даних
- У методі `loadDocumentAs` кешувати результати успішного зчитування файлу та парсингу (наприклад, результат `loadNANAsync` або `loadYAMLAsync`).
- Зберігати результат у `this._dataCache` за ключем `uri`.

### 4. Інвалідація Кешу (Cache Invalidation)
- У методах `saveDocument`, `writeDocument`, `dropDocument` обов'язково очищати кеш для зміненого `uri` (або просто скидати відповідні ключі у `Map`).
- Опціонально: додати метод `clearCache()`, щоб мати можливість повністю скинути пам'ять, коли це необхідно.

### 5. Покриття Тестами
- Написати юніт-тести в `packages/db-fs/src/DBFS.test.js`, які перевірятимуть:
  1. Перший виклик `db.fetch()` читає файл (I/O).
  2. Другий виклик того ж `db.fetch()` НЕ робить системного виклику (mock `FS.stat` або перевірка швидкості).
  3. Виклик `saveDocument` очищає кеш, і наступний `fetch` знову читає з диска.
  4. Швидкість зчитування масиву з сотні файлів (performance тест).

## Кроки для ручної реалізації

1. **Модифікація конструктора:** Відкрий `packages/db/src/DB.js` (або `db-fs/src/DBFS.js`) та додай властивості `_statCache = new Map()` та `_dataCache = new Map()`.
2. **Оновлення методів читання:** Оберни логіку в `statDocument` та `loadDocument` перевіркою `if (this._statCache.has(uri)) return this._statCache.get(uri)`.
3. **Оновлення методів запису:** У `saveDocumentAs`, `dropDocument` додай `this._statCache.delete(uri)`.
4. **Тестування:** Запусти тест `npm run test:all` або `performance.test.js`, щоб побачити, як час обробки 60 файлів падає з 600мс до ~30мс.
