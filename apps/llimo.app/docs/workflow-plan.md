# LLiMo Workflow Implementation Plan

This document outlines the detailed plan to implement the autonomous workflow engine and proxy command system in the `llimo.app` codebase, based on our successful experiments in `llimo.app-dev` and the new Model-as-Schema architecture.

## 1. Goal
Introduce an advanced Workflow Execution Pipeline to `llimo.app` that allows autonomous, self-correcting task execution using LLiMo's Language Living Models. The agent will read a workflow, generate code, intercept tests, and refine its solution if tests fail, all while operating inside a safe execution boundary. Everything should be described using the `Model-as-Schema` pattern.

## 2. Core Components to Migrate/Implement

### A. Workflow Parser and Validator
- **Purpose**: Parse markdown files to understand the workflow and validate steps.
- **Features**:
  - Use the `markdown` package to parse the workflow definitions into distinct, readable blocks.
  - Establish a verification step after every code block execution (e.g., automatically validate all generated JS files for syntax errors since models like Groq can occasionally return broken files).

### B. SecurityGate (`src/workflow/SecurityGate.js`)
- **Purpose**: A gatekeeper for any system commands generated by the LLM.
- **Features**:
  - `ALLOWED_COMMANDS`: strict allow-list including `pnpm`, `npm`, `node`, `git`, `ls`, `cat`, `echo`, `pwd`, `grep`, `find`, `sed`, `awk`, `head`, `tail`, `wc`.
  - `FORBIDDEN_PATTERNS`: Regex patterns blocking malicious execution (e.g., `/rm\s+-rf\s+\//`, `/sudo/`, `/eval/`).
  - `validate()`: Rejects forbidden actions and approves explicitly allowed commands or `@` proxy tools.

### C. ProxyRegistry (`src/workflow/ProxyRegistry.js`)
- **Purpose**: Creates an internal tool-proxy layer for agent-specific operations without polluting standard bash context.
- **Features**:
  - Commands starting with `@` (e.g., `@validate`, `@bash`, `@ls`, `@get`, `@web`) map to inner JavaScript functions instead of CLI sub-processes.
  - Transparent execution for the AI: AI just uses markdown bash blocks, but `WorkflowRunner` dispatches to `ProxyRegistry` if it's an internal command.

### D. Web Extraction Tool (`@web` or similar)
- **Purpose**: Allow the workflow engine to pull external text content.
- **Features**:
  - A headless browser tool (using `wget`, `playwright`, or `chrome`) that is accessible to the agent.
  - The tool should fetch pure text or HTML-to-text to feed into the LLM context.

### E. TestRunner (`src/workflow/TestRunner.js`)
- **Purpose**: Executes `pnpm test` and extracts meaningful diagnostics using the TAP protocol.
- **Features**:
  - Uses `@nan0web/test` tools (`runSpawn`, `TapParser`) to parse TAP output into structured errors.
  - Feeds exact failure reasons (e.g., `file.js:15 - Expected true to be false`) back into the context for self-correction.

### F. WorkflowRunner Update (`src/workflow/WorkflowRunner.js`)
- **Purpose**: Orchestrates the retry loop.
- **Features**:
  - Validates `FileProtocol.validate()` generated changes against local FS.
  - Tries up to `N` attempts to execute the workflow strategy, run `TestRunner`, parse failures, append them to the prompt, and retry on failure.
  - Enforces `SecurityGate` on every parsed command before execution.
  - Branches tool execution to `ProxyRegistry` vs standard `BashCommand`.
  - **Budget Tracker**: Track the expenses (spend limit per workflow) locally preventing cost overruns.

### G. Model-as-Schema Transitions
- All components, runners, and logic definitions should follow the `Model-as-Schema` workflow (`@/model-schema`).
- Instead of loose configurations, everything (e.g., cost limits, commands, UI prompts) should be defined completely within `static` blocks in classes, while instances represent runtime state. No ad-hoc validation outside the classes.

## 3. The New Architecture Shift (App Pipeline)
As planned, before or during these implementations, run the **App Pipeline** to upgrade `llimo.app` to the new architecture.

**Key Requirements:**
- **UI CLI Transition**: Replace the old terminal logging and prompting with `@nan0web/ui-cli` (interactive components, TAB-search, Spinner, ProgressBar async controller).
- **Zero-Hallucination Framework**: Use strict `Model-as-Schema` parsing when building out the proxy options or handling responses from OpenAI/Cerebras APIs.
- **Unified Engine**: Move the logic layer towards our robust patterns used across other elements (like Data-Driven UI and `resolveValidation` loops) while stripping outdated components.

## 4. Execution Steps
1. Delete `llimo.app-dev`.
2. Run the `app-pipeline` workflow to adapt `llimo.app` to the new architecture (`UI CLI`, `Model-as-Schema`, etc).
3. Introduce the `src/workflow/` directory.
4. Implement `SecurityGate`, `ProxyRegistry`, `TestRunner`, test validators, and the overarching `WorkflowRunner` equipped with spend tracking and markdown blocking.
5. Integrate these deeply into LLiMo pipeline so running tasks can be 100% automated with recursive self-fixing.
