# 🧠 Стратегія AI LLiMo (AI Strategy Plan) - Докладна версія

На основі аналізу пакетів `@nan0web/ai` та `@nan0web/llimo`, ми розробили план вдосконалення інтелектуального вибору моделей.

## 1. Аналіз поточного стану
Ми виявили клас `AiStrategy` у пакеті `ai` (src/domain/AI.js) та локально у `llimo.app`. Він вже підтримує базові параметри:
- **Finance:** `free`, `cheap`, `expensive`.
- **Speed:** `slow`, `fast`.
- **Volume:** `low`, `mid`, `high` (кількість параметрів моделі).
- **Level:** `simple`, `smart`, `expert` (на основі статистики успішності).

## 2. План розширення та оптимізації

### А. Вимірювання та Репутація (Real-time Stats)
- **Швидкість (Performance):** Реалізувати вимірювання `tokens/sec` під час генерації. Це дозволить стратегії `speed: fast` реально обирати найшвидшу модель на основі останніх запитів користувача.
- **Якість (Quality Audit):** Інтегрувати автоматичну валідацію відповідей. Якщо модель генерує синтаксично неправильний код (broken JS/YAML), це автоматично знижує її `reputation score`.

### Б. Динамічне перемикання (Self-Correction Loop)
- **Fallback Logic:** Якщо поточна модель (напр. "cheap/fast") видає помилку валідації, система автоматично перемикається на "expert" модель для виправлення кроку.
- **Yaml Strategy:** Можливість задавати стратегію безпосередньо в MD-файлах завдань:
  ```yaml
  strategy:
    finance: cheap
    level: expert
  ```

### В. Глобальна статистика (Home DB)
- **Централізоване сховище:** Всі логи зберігаються у `~/.llimo/chats/`.
- **Аналіз витрат:** Глобальний файл `index.csv` дозволяє бачити сумарні витрати та ефективність по кожному провайдеру (Groq, Gemini, DeepSeek).

## 3. Кроки впровадження (Action Items)

1. **Модернізація `Usage.js` та `AI.js`:** Додати поля для затримок та швидкості.
2. **Оновлення `chatProgress.js`:** Відображення живої статистики швидкості та залишку безкоштовних лімітів (RPD/TPD).
3. **Реалізація `QualityGate.js`:** Скрипт-валідатор для згенерованого коду (eslint-lite / syntax check).
4. **Інтеграція в `WorkflowRunner`:** Автоматичне використання "Expert" моделей при виявленні "broken" файлів.
