# LLiMo App: Що Зроблено та Куди Рухаємось

Цей документ фіксує поточний стан розробки `llimo.app` після переходу на нову OLMUI (Zero-Hallucination) архітектуру та визначає наступні кроки.

## ✅ Що вже зроблено (Квітень 2026)

1. **Затверджено "Constitution" (Крок 1):**
   - Написано ідеальний `seed.md` (базова мета, сутності, обмеження).
   - Сформовано 25 User Stories у `docs/user-stories.md`, які описують поведінку: CLI, парсинг, Safety Gate (блокування `bash rm -rf /`), трансляція агностичних команд `@llimo`, логування, CSV індексація.

2. **Затверджено Архітектуру Монетизації та IP Захисту:**
   - Оновлено `project.md`. LLiMo є "тупим тунелем-рабом", який виконує проксі команди (`@llimo build`). Самі цінні пайплайни ("Золоті Стандарти", напр. `/release`) і промпти закриті від користувача і лежать на `api.llimo.app`. Безкоштовний тільки базовий AI-чат (BYOK або Local OS).

3. **Створені Доменні Моделі (Model-as-Schema) - (Крок 2):**
   - `WorkflowModel.js`: Головний оркестратор. Тепер має поле `templateURL` для завантаження SaaS-пайплайнів. Метод `async *run()` реалізує цикл **AI Self-Correction** (передає логи назад до AI при падінні команди, робить retry, а не вилітає з Node.js).
   - `WorkflowStepModel.js`: Описи кожної проксі-інструкції (валідні тільки `@llimo`, `@bash`, `@web`, тощо).
   - `ChatSessionModel.js`: Управління станом сесії. Логи розведено по папках `~/.llimo/chats/YYYY/MM/DD/chatId/`.
   - `LLiMoConfigModel.js`: Локальний конфіг (провайдер, ліміти витрат).

4. **Гігієна коду:**
   - В усіх моделях прописана повна JSDoc-типізація (`/** @type {string} */`) разом із `resolveDefaults()`.
   - Написано перший сценарний In-Memory Unit Тест (`WorkflowModel.story.js`), який перевіряє User Stories #18/19 (цикл самовиправлення при помилці).
   - Всі терміни (`@nan0`) в межах програми `llimo.app` замінено на канонічний `@llimo` (а в логах CLI він малюється як кольоровий фіолетовий @ll∙mo). Для `C++` / `Rust` прибрано `N/A`, додані реальні драйвери команд (ctest, cargo test --doc).
   - Додана інструкція `workflows/docs-stories.md` для написання сценаріїв.

## 🚀 Наступні Кроки (Що Робимо Далі)

### 1. `/pipeline-no3-contract`: Контрактне тестування
Ми написали один `.story.js` для `WorkflowModel`. Але тепер треба:
- Перекласти інші важливі User Stories з `docs/user-stories.md` у `*.story.js` тести з In-Memory `DB`.
- Написати специфікації-контракти для CLI Adapter (`WorkflowCLIAdapter.story.js`), щоб впевнитись, що він правильно малює `ll∙mo`, парсить TAP `Pass/Fail` з тестів і обриває `stdout` для Alert при падінні.

### 2. `/pipeline-no4-adapter`: Рефакторинг UI Адаптерів 
- Старий `src/workflow/WorkflowCLIAdapter.js` працює як один великий монолітний While-цикл.
- Його треба переписати так, щоб він став **Споживачем DTO інтенцій генератора** (`for await (const intent of new WorkflowModel(data).run()) { ... }`).
- Додати підтримку Cloud Template API (якщо є `templateURL`, качати і шифрувати в пам'яті).
- Імплементувати "тупий" CLI-чат (`llimo chat`), який просто взаємодіє з LLM (а-ля BYOK), за архітектурою схожою на те що описано в старих архівах (`bin/llimo-chat.js`), але через OLMUI.

### 3. Автоматизація Регресії (`/release`)
- Навчитись запускати команду `@llimo test:stories` по всьому репозиторію. У нас з'явився новий формат релізів (міграція `releases/**/*.spec.js -> src/test/releases/**/*.test.js`), ми маємо інтегрувати цю структуру в `LLiMo.app` і написати для неї CLI-команду `llimo release:close` або зробити alias в NPM (`pnpm run release:close`).
