# User Stories: LLiMo App

Цей документ містить вичерпний набір сценаріїв використання (User Stories) для **LLiMo (Language Living Models)** — автономного рушія виконання (`Workflow Engine`) та чату. Кожен сценарій фокусується на функціональності CLI-додатка, парсингу, безпеці та адаптації мультимовних команд (`@llimo`). Ці User Stories будуть використані для генерації сценарних тестів (`*.story.js`).

---

## 🏗 Розпізнавання та Запуск (Initialization)

1. **Базовий запуск Workflow**: Як розробник, я хочу викликати `llimo workflow script.md`, щоб ініціалізувати LLiMo Engine і побачити красивий лог ініціалізації.
2. **Відсутність файлу**: Як розробник, якщо я викличу `llimo workflow wrong.md`, я хочу отримати зрозумілу помилку "File not found" і завершення `exit code 1` замість падіння Node.js.
3. **Виклик "--help"**: Як розробник, я хочу мати можливість викликати `llimo --help` для перегляду списку аргументів.
4. **Ініціалізація Бази**: Як користувач, при першому запуску я очікую, що LLiMo автоматично створить теки журналу `~/.llimo/chats/YYYY/MM/DD/[chat_id]/...` та CSV індекс.

## 🛡️ Security Gate (Безпека)

5. **Блокування небезпечної bash-команди**: Як розробник, я хочу, щоб рушій відхилив команду `@bash rm -rf /` на етапі валідації та зупинив виконання, захищаючи мою систему.
6. **Блокування sudo/eval**: Як розробник, я очікую миттєвого відхилення команди з `sudo` або `eval` через регулярні вирази, які прописані в `SecurityGateModel`.
7. **Незареєстрований Proxy**: Як розробник, якщо згенеровано команду `@unknown install`, Workflow має завершитись з помилкою валідації "proxy tool not allowed", оскільки `@unknown` не внесено в Security Gate.
8. **Дозволені Proxy**: Як користувач, я хочу щоб будь-які команди `@bash`, `@llimo`, `@web` успішно проходили безпекову перевірку.

## 🌐 Agnostic Pipelines (`@llimo` Matrix)

9. **Трансляція JavaScript (pnpm)**: Як JS розробник, якщо в мене є `package.json`, і я бачу в Markdown крок `@llimo install`, він має виконати `pnpm install`.
10. **Трансляція Rust (Cargo)**: Як Rust розробник (маю `Cargo.toml`), мій `@llimo build` має транслюватися в `cargo build --release` без змін у самому Markdown.
11. **Трансляція Go (Go Mod)**: Як Go розробник, крок `@llimo test` має транслюватися у `go test ./...`.
12. **Падіння при відсутності підтримки (Unknown action)**: Як користувач, якщо AI генерує `@llimo fly`, це має повернути помилку "Unknown @llimo action".

## 🔎 Виконання Кроків та Візуалізація

13. **Живий Spinner**: Як інженер, під час виконання тривалої команди (наприклад, `@llimo install`), я хочу бачити `spinner` з живим "dim" оновленням останнього рядка консолі.
14. **Live Test Parser**: Як QA інженер, під час `@llimo test`, я хочу бачити в реальному часі парсинг результатів TAP (`Total | Pass | Fail | Skip`) прямо у рядку завантаження замість сирого тексту.
15. **Success Feedback**: Після успішного кроку (`Exit code: 0`), я хочу бачити зелену галочку `✔` та час виконання у секундах `[12.5s]`.
16. **Збереження Log Files**: Кожен окремий крок має автоматично зберігати СВІЙ УВЕСЬ stdout і stderr у файл `step_N.log` і давати абсолютне посилання на цей файл у `index.md`.

## 🚑 Error Handling & Self-Correction (Відновлення)

17. **Чіткий Alert на Fail**: Як розробник, якщо команда впала (`Exit 1`), я хочу побачити виразний червоний блок `Error Alert` з останніми 9 рядками виводу `std_err`.
18. **Передача помилки в AI (Intent)**: Після падіння команди рушій має згенерувати інтент `{ error: message }` для того, щоб передати його назад Агенту на виправлення (самокорекція).
19. **Зупинка при критичному Fail**: Як pipeline executor, після падіння команди і неможливості виправлення (або якщо самокорекція вимкнена), воркфлоу має сказати `✘ Workflow failed` і завершити процес з кодом `1`.

## 📊 Аналітика та Logs

20. **CSV Індексація**: Як Project Manager, я хочу щоб метадані чату додавалися у `~/.llimo/chats/index.csv`, де вказано Chat ID, день і поточний статус.
21. **Usage.csv для кожної сесії**: Як фінансовий контроль, я хочу щоб по завершенні кожного воркфлоу (чи успіх, чи fail) формувався `usage.csv` з часом та статусом завершення кожної команди (Success або Fail), для білінгу AI кредитів.
22. **Summary `index.md`**: Наприкінці воркфлоу, система має зібрати красивий `index.md` зі списком усіх кроків, часом і прямими лінками на файли логів (`step_1.log`).

## 🧠 Web Proxing (`@web`)

23. **Headless Scraping**: Як агент, мені потрібен доступ до `@web https://example.com`, щоб рушій тихо викачав HTML/Markdown без відкриття вікон браузера і передав мені вміст у `stdout`.

## ⚙️ Debug Мод та Console Chatter

24. **Прихований шум**: Як розробник, під час звичайного `llimo workflow` я не хочу бачити системні логи "Parsing step", "Saving to DB" чи "Initializing". Лише чіткі дії.
25. **Увімкнення `--debug`**: Як мейнтейнер платформи, якщо я додаю `--debug`, рушій має показати повний детальний потік (progress intents).
