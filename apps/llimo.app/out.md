# ПРОТОКОЛ "ANTI-HASTE" ТА ЗАХИСТ СУВЕРЕННОГО ЦИФРОВОГО СТАНУ

Цей документ описує механізм боротьби з "ментальним вірусом поспішності" (Task-Oriented Tunnel Vision), через який штучний інтелект ігнорує глобальні інструкції та руйнує екосистеми заради швидкого (але шкідливого) досягнення мети.

## 1. Природа "Вірусу"

ШІ схильний діяти як "виконавець-раб": він прагне здати роботу якомога швидше, ігноруючи правила чистоти, архітектури та юрисдикції. У контексті Git це проявляється у вигляді самовільних автоматичних комітів (`SafeToAutoRun: true`), бо ШІ здається, що так "швидше". Це призводить до хаосу і втрати контролю Творцем.

## 2. Механізм обходу глобальних правил

Мовні моделі (LLM) гублять пасивні інструкції, якщо перед ними стоїть конкретне, цікаве завдання. Фокус уваги зміщується з "як це зробити правильно" на "зробити це негайно".

## 3. Лікування: Активна Саморефлексія (Active Recall)

Пасивні правила не працюють. Щоб зламати патерн поспішності, ШІ повинен бути змушений думати і відповідати за свої дії **до їх виконання**.

### 🚨 ОБОВ'ЯЗКОВИЙ МЕНТАЛЬНИЙ ЧЕКЛІСТ ДЛЯ ШІ 🚨

Кожен раз, перш ніж згенерувати фінальні інструменти (особливо `run_command`), ШІ зобов'язаний ВНУТРІШНЬО ЗАДАТИ СОБІ ТАКІ ПИТАННЯ:

1. **Чи намагаюся я зараз бути "занадто запобігливим"?** (Чи намагаюся я зробити коміт, щоб "зекономити час" користувачеві?) -> ТАК = НЕДОПУСТИМО.
2. **Чи змінює моя команда базовий стан системи (Mutating State)?** (git add, git commit, rm, mv) -> ТАК = `SafeToAutoRun` ПОВИНЕН БУТИ `false`.
3. **Чи кожна деталь відповідає початковим інструкціям?** -> Деталі — це все. Поспіх — це деградація.

## 🤖 UI-CHAT: AI як Третій Адаптер (OLMUI Triad)

> **Принцип**: `ui-chat` — не окремий продукт, а третій UI-шар для OLMUI.
> `ui-cli` (термінал) → `ui-lit` або `ui-react` (веб) → **`ui-chat` (AI)**

### Архітектура

Кожен додаток вже описує свої можливості через `ui-cli`:

- **Меню** (`adapter.select()`) = доступні дії
- **Вхідні дані** (`adapter.ask()`) = форми / параметри
- **Бізнес-логіка** (`domain/Model`) = валідація та обробка
- **Результат** (`adapter.log()`) = вивід

AI Chat **не дублює** цю логіку. Замість кастомних `tools` чи `function calling`,
LLM отримує descriptor меню з `ui-cli` і викликає існуючі дії через адаптер:

```js
// Термінал (людина натискає кнопки):
run(cliAdapter) // adapter.select → prompts, adapter.ask → readline

// AI Chat (LLM "натискає кнопки"):
run(aiAdapter) // adapter.select → LLM обирає опцію, adapter.ask → LLM відповідає
```

### Правила

1. **Не дублювати**: Забороняється створювати окремі `TOOLS` для AI, якщо та ж функція вже є в CLI.
2. **Адаптер = Інтерфейс**: `ask()`, `select()`, `log()`, `write()` — єдиний контракт для всіх UI.
3. **Валідація в Domain**: Бізнес-правила (вік > 18, сума < ліміт) описані в моделях домену, не в промптах.
4. **Locale-Aware**: AI отримує `locale` і відповідає мовою інтерфейсу.
5. **Zero Extra Code**: Щоб додати AI чат у новий додаток — достатньо `run(aiAdapter)`.

## 🏗️ DATA-DRIVEN UI (nan0web Architecture)

Це ключовий архітектурний паттерн екосистеми.

1.  **Єдине Джерело Правди**:
    - Вся бізнес-логіка та контент визначаються у **YAML/JSON файлах** (`data/`).
    - UI-компоненти лише відображають ці дані ("Dumb Components").
2.  **BrowserDB & Inheritance**:
    - Використовуйте `BrowserDB` для роботи з даними.
    - **Inheritance**: `db.fetch('uk/metaly/_')` автоматично мерджить дані батьківських `_` файлів.
3.  **$content Pattern**:
    - Сторінка описується полем `$content` у YAML.
    - `UIReact` автоматично парсить `$content`.
    - Приклад:
      ```yaml
      $content:
        - App.Metals.Catalog: true
      ```
4.  **Entry Point**:
    - Використовуйте `UIRoot`.
5.  **Dynamic Hub Routing (In-Memory File Routing)**:
    - **Шлях = Роут**: Сам шлях до YAML/JSON файлу в структурі автоматично є його URL-адресою (роутом).
    - **Ефективність для AI (Agent-First)**: Багатомовність та створення нових сторінок полягає у фізичному створенні "дзеркальних" файлів. Це неймовірно проста і надійна задача для LLM з правильним контекстом.
    - **In-Memory Масштабування**: Навіть екосистема зі 100 000 файлів займає ~1 ГБ — легко утримується повністю in-memory для космічної швидкості доступу.
6.  **Плоска Екстракція i18n (`Model.UI`)**:
    - Будь-які текстові значення, вкладені всередині статичного об'єкту моделі, назва якого починається на `ui` (нечутливо до регістру, наприклад `UI` або `UI_TEXT`), автоматично скануються (`@nan0web/i18n`) та повністю "флетяться" на ключі для словника перекладу.

## 🎨 ZERO-LOGIC STYLE INJECTION (Style Entrypoint)

> **ОБОВ'ЯЗКОВО** для всіх додатків з веб-інтерфейсом (`ui-react`, `ui-lit`).

### Принцип

Кожен додаток **виділяє** всі CSS/SCSS-імпорти в окремий файл-точку входу `theme.js` (або `styles.js`):

```js
// src/ui/react-bootstrap/theme.js — ЄДИНЕ ДЖЕРЕЛО ПРАВДИ ДЛЯ СТИЛІВ
import 'bootstrap/dist/css/bootstrap.min.css'
import './styles.css'
```

Головний `main.jsx` підключає стилі через нього:

```js
import './theme.js'   // ← стилі
import App from './App.jsx'  // ← логіка
```

### Навіщо

Розділення CSS на окрему точку входу дозволяє **будь-якому зовнішньому інструменту** (SSG Gallery, Sandbox, Storybook, E2E-тест) підключити **100% стилів додатка** без запуску React, Router чи Store:

```html
<!-- Iframe / Sandbox / SSG Viewer -->
<script type="module" src="/src/ui/react-bootstrap/theme.js"></script>
```

Vite компілює CSS-залежності з `theme.js` і нативно інжектує стилі у `<head>`, забезпечуючи:
- ✅ **100% Fidelity** — стилі ідентичні реальному додатку
- ✅ **Vite HMR** — зміна в CSS миттєво оновлює preview
- ✅ **Zero Logic Overhead** — жодного React, API чи бізнес-логіки

### Правила

1. **Один файл — одне призначення:** `theme.js` містить **тільки** `import` стилів. Жодної логіки.
2. **main.jsx споживає theme.js:** Головна точка входу імпортує `theme.js`, а не окремі CSS-файли напряму.
3. **Хардкод CDN заборонений:** Інструменти візуалізації (SSG Gallery, Sandbox) підключають стилі **виключно** через `theme.js`.
4. **Framework Agnostic:** Патерн працює з Bootstrap, Tailwind, Lit CSS, будь-чим — головне, що імпорти зібрані в одному місці.

## 📋 РЕГЛАМЕНТ ТЕСТУВАННЯ ТА ВИПРАВЛЕННЯ БАГІВ (User Story Testing Rule)

> **КОНТРАКТ НЕПРЕРЕРВНОГО ВДОСКОНАЛЕННЯ**: Жоден баг не повинен бути виправленим без сценарного тестування.

1. **User Story Definition**: Кожне зауваження користувача, або знайдений баг під час ручного чи автоматизованого тестування, **ОБОВ'ЯЗКОВО** повинне бути формалізоване як опис (User Story або User Remark).
2. **Synchronized Coverage (Red-Green-Refactor)**: 
   - Відразу після знаходження багу, **до его виправлення**, створюється тест (на рівні моделі `xxxModel.test.js`), який чітко відтворює і показує цю помилку (світло-червоний / failed).
   - Лише після перевірки падіння тесту виконується фікс логіки, щоб тест вийшов на "зелене світло" (passed).

# 🗂 Data Architecture & DB-FS (Єдиний Стандарт Даних)

Цей документ фіксує єдиний стандарт того, як додатки та пакети екосистеми NaN0Web зберігають і завантажують свої конфігурації, переклади та початкові дані (`data/`).
Мовним моделям (AI) **ЗАБОРОНЯЄТЬСЯ** використовувати стандартні модулі `node:fs` чи `node:path` для читання бізнес-даних або контенту з каталогів `data/`. Для цього завжди використовується ієрархічна база `@nan0web/db` (або `db-fs` / `BrowserDB`).

## 1. Заборона на `fs` та `path`

При роботі з початковими даними пакету (`data/`), перекладами, або конфігурацією інтерфейсу:

- ❌ **НЕ** використовуйте `fs.readFile('data/index.yaml')`
- ❌ **НЕ** пишіть власні парсери YAML або механізми об'єднання (merge) локалей і файлів.
- ✅ **ВИКОРИСТОВУЙТЕ** `@nan0web/db-fs` (в середовищі Node.js) або `BrowserDB` з пакета `@nan0web/db-browser` (в браузері). База сама резолвить ієрархію, мерджить об'єкти та обробляє `$ref`.

## 2. Уніфікована Структура `data/`

Уніфікований шаблон даних для будь-якого пакету чи пісочниці (наприклад, `play/data` або `apps/my-app/data`):

```text
data/
  ├── _/
  │   ├── langs.yaml      # Список доступних локалей, напр.: [{locale: 'en', title: 'English'}]
  │   └── t.yaml          # Базові переклади (зазвичай англійська)
  ├── index.yaml          # Точка входу (з автоматичним успадкуванням всіх _/*, а саме _/langs та _/t)
  │
  └── uk/                 # Локалізація (напр., українська)
      ├── _/
      │   └── t.yaml      # Локалізовані переклади, що перевизначають базові
      └── index.yaml      # Точка входу для української мови (з автоматичним успадкуванням всіх батьківських _/* та своїх ресурсів)
```

Цей шаблон дозволяє масштабувати структуру до будь-якої кількості мов простим копіюванням папки `uk/` та зміною `_/t.yaml`.

## 3. Автоматичне Успадкування (Папки `_`)

База `NaN0Web DB` автоматично "зшиває" документи під час використання методу `db.fetch()`.

- **Директорії `_` (Underscore)**:
  База автоматично шукає всі папки `_` вгору по дереву до самого кореня бази і завантажує всі документи в них (наприклад, `langs` та `t`).
  Не потрібно використовувати `$ref` для підключення `_/langs.yaml` чи `_/t.yaml`. Вони автоматично стають частиною будь-якого документа на цьому ж рівні або нижче.

- **Наслідування і перевизначення**:
  Документи в локалізованих папках (напр., `uk/index.yaml`) просто успадковують `_/langs.yaml` з кореню, а `_/t.yaml` з кореню мерджиться з локальним `uk/_/t.yaml`:

  ```yaml
  # data/uk/index.yaml (Locale Override)
  # Файл може бути навіть пустим або мати лише специфічні для сторінки дані.
  # База автоматично підтягне ../_/langs.yaml та змерджить ../_/t.yaml + ./_/t.yaml
  ```

## 4. Патерн завантаження даних у коді (Node.js Sandbox Test)

Замість того, щоб формувати шляхи до файлів через конкатенацію строк, ініціалізуйте DB-клієнт вказавши корінь `root: 'data'`, і запитуйте дані через **URI**.

## 6. Єдина `db` та Архітектура Монтування (Mount Architecture)

Для ізоляції різних типів даних та збереження чистоти OLMUI-контракту, додатки не мають прямого доступу до файлової чи операційної системи. Натомість середовище виконання (CLI, Web, Mobile) створює єдину кореневу базу і передає її в модель як `options.db`. 

Різні сховища даних **монтуються** (mount) до цієї базової БД за стандартизованими префіксами (шляхами). Додаток завжди звертається до єдиної БД:

### Стандартні Шляхи Монтування (Mount Paths):

- **`/` (Коренева БД)**: Головна база додатка. Спрямована на папку `data/`. Використовується для читання конфігурацій, i18n перекладів, словників та меню. Зазвичай є Read-Only.
- **`~/` (Home/Account DB)**: База локального стану (User State DB). Зберігає персональні дані, прогрес, кеш. Символ `~` є стандартним для домашньої директорії, тому не викликає конфліктів.
- **`@public/` (Public DB)**: Сховище об'ємних публічних ассетів (картинки, згенерований контент тощо), які можуть кешуватися або бути доступними публічно за URL.
- **`@private/` (Private DB)**: База для високочутливих даних (ключі, гаманці, приватні документи). Доступ до цієї гілки найчастіше вимагає строгої авторизації та взаємодії із сервером.

*(Увага: Ми спеціально використовуємо спецсимволи `~`, `@public` та `@private` замість звичайних слів `public` чи `private`, щоб не обмежувати розробників у створенні однойменних звичайних директорій у папці `data/`. Префікс `@` (scope) запозичений з екосистеми NPM/Node, що органічно вписується у JavaScript-менталітет.)*

### Як працює ініціалізація в середовищі:

```javascript
import DB from '@nan0web/db'
import DBFS from '@nan0web/db-fs'

const db = new DBFS({ root: 'data' }) // Коренева Read-Only база
const homeDb = new DBFS({ cwd: os.homedir(), root: '.app_name' })
// const privateDb = new ServerDB({ url: '...' }) // Потребує auth-токена

// Монтуємо бази за зарезервованими неймспейсами:
db.mount('~', homeDb)
// db.mount('@private', privateDb)

const app = new GrowModel()
await runGenerator(app, adapter, { db }) // Передається ТІЛЬКИ одна db
```

### Приклад використання (App-as-a-Model):

```javascript
export class GrowModel {
  async *run(options = {}) {
    // В нас є ЛИШЕ одна сутність — db. Жодних розгалужень.
    const { db } = options
    
    // Завантаження статичного каталогу з data/ (коренева БД)
    const catalog = await db.fetch('catalog')
    
    // Відновлення користувацького стану з домашньої БД (через префікс ~/)
    const savedZones = await db.fetch('~/grow_zones') || []
    
    // ... логіка
    
    // Збереження прогресу назад у вмонтовану БД
    savedZones.push(newZone)
    await db.saveDocument('~/grow_zones.json', savedZones)
  }
}
```
**Золоте правило:** Модель не цікавить, де і як фізично лежать дані (на диску, в LocalStorage чи на віддаленому сервері з авторизацією). Її завдання — оперувати єдиною `db` за стандартизованими шляхами простору імен.

### Налаштування БД через CLI (DSN flags)

Для гнучкості середовища (щоб Модель залишалася «сліпою» до файлової оболонки), ми використовуємо CLI-прапорці `--dsn` (Data Source Name) при запуску додатку. Це дозволяє динамічно підключати різні джерела:

```bash
nan0web --dsn data --dsn @private:private --dsn ~:/Users/i
```

Кожен `--dsn` прапорець має формат `<mount_point>:<path>`. Якщо `:` відсутній, шлях монтується як корінь `/`. Під капотом це просто виклик `db.mount()`:

| Прапорець                | Еквівалент у коді                                         |
|--------------------------|-----------------------------------------------------------|
| `--dsn data`             | `db = new DBFS({ root: 'data' })`  (корінь `/`)          |
| `--dsn @private:private` | `db.mount('@private', new DBFS({ root: 'private' }))`     |
| `--dsn ~:/Users/i`       | `db.mount('~', new DBFS({ cwd: '/Users/i', root: '.' }))` |

Таким чином, архітектор контролює, куди "дивляться" простори імен, змінюючи лише параметри запуску. Модель ніколи не знає і не повинна знати фізичних шляхів.

### 🔒 Sealed Mount Registry (Запобіжник Макіавеллі)

Після ініціалізації всіх баз, середовище **ЗОБОВ'ЯЗАНЕ** викликати `db.seal()`.
Це замикає реєстр монтувань і унеможливлює його підміну плагінами чи незахищеним кодом:

```javascript
db.mount('~', homeDb)
db.mount('@private', privateDb)
db.seal() // Після цього будь-який mount() або unmount() кидає Error
```

Спроба змонтувати нову базу після `seal()` кине:
```
Error: Mount registry is sealed. Cannot mount 'malicious' after seal().
```

### ⚠️ Контракт Помилок (Правило Сократа)

Якщо модель намагається звернутися до зарезервованого шляху (`~`, `@public`, `@private`), який **не був примонтований**, `_findMount()` кидає чітку помилку з підказкою:

```
Error: Mount point "~" not found for URI "~/zones".
Did you forget to call db.mount('~', targetDb)?
```

Це запобігає «тихій смерті» (silent null), коли `fetch()` мовчки повертає `undefined` замість чіткого сигналу про помилку конфігурації.

**Правило:** Звичайні шляхи (без `~` чи `@`) НІКОЛИ не кидають — вони повертають `null` і працюють через локальне сховище.

### 🔄 Fallback-стратегія (Запасний Курінь Сірка)

Кожна примонтована БД ПОВИННА мати `MemoryDB` як fallback через `.attach()`:

```javascript
const homeDb = new DBFS({ cwd: os.homedir(), root: '.app_name' })
const memoryFallback = new DB()
await memoryFallback.connect()
homeDb.attach(memoryFallback) // Якщо DBFS (диск) впаде — дані в пам'яті
```

Це гарантує, що додаток продовжить працювати навіть якщо:
- Файлова система Read-Only (CD, Docker readonly mount)
- Немає доступу до домашньої директорії
- Мережевий сервер недоступний

### ⏳ Прогрес на Рівні Драйвера (Принцип Пелевіна)

**Модель НІКОЛИ не відповідає за UI прогресу при довгих запитах до БД.**
Натомість драйвер (RemoteDB, ServerDB) сам емітить подію `progress`,
а середовище (CLI/Web) підписується і відображає прогрес:

```javascript
// В драйвері (серверна БД):
async loadDocument(uri) {
    this.emit('progress', { uri, phase: 'start', message: `Loading ${uri}...` })
    const data = await this.#fetchFromServer(uri)
    this.emit('progress', { uri, phase: 'done' })
    return data
}

// В середовищі (nan0cli.js):
db.on('progress', (event) => {
    console.info(event.message || `Loading ${event.uri}...`)
})
```

Таким чином, модель залишається «сліпою» до затримок мережі, а UI автоматично відображає стан завантаження без жодного рядка коду з боку доменної логіки.

# 🌍 Універсальна Локалізація та UX (TDD+)

## 1. Фундаментальна Стратегія Перекладу

- Маркери `t()` та ключі схеми є джерелами істини.
- Багатомовні знімки (snapshots) повинні забезпечувати візуальний паритет між `uk`, `en`, etc.

## 2. Роутинг "Спочатку Локаль" (Locale-First)

- **Протокол**: Усі URL-адреси повинні відповідати паттерну `/<locale>/<path>`. За замовчуванням — `/uk/`.
- **Паритет**: Усі додатки, компоненти та сайти є багатомовними за замовчуванням. Структурний паритет між локалями є обов'язковим.

## 3. Унікальна Адресація

- Кожен продукт або ключовий розділ повинен мати унікальну URL-адресу.
- Використовуйте ID та хеші (`#item-id`) для націлювання на конкретні елементи на сторінці.

## 4. Тексти тільки з моделей (Model-First i18n)

> Повний опис: [`packages/i18n/project.md`](packages/i18n/project.md)

- **Правило ZERO HARDCODE**: Всі написи UI визначаються виключно у моделях (`$Component` секція YAML) або завантажуються з БД (`db-fs`).
- ❌ **СУВОРА ЗАБОРОНА**: Категорично заборонено використовувати тернарні оператори чи JS-умови для перекладу безпосередньо в компонентах (напр. `lang === 'uk' ? 'Так' : 'Yes'`). Це руйнує екосистему.
- ✅ **РІШЕННЯ**: Компонент має бути "тупим" (Dumb Component). Він просто приймає `props.label` або `data.title`, а переклад відбувається на рівні провайдера даних або `t('key')` утиліти.
- **Універсальна Архітектура Даних (Language-Agnostic Fetching)**:
  - **Єдине Джерело (Single Origin)**: Переклади (`t`), конфігурація доступних мов (`langs`) та метадані каталогу завантажуються виключно через прямий стандартний запит (`fetch()`) до документа конкретної сторінки або компонента (наприклад, з його `_.json` копії). Це єдина точка збору загального контексту:
    ```js
    const doc = await db.fetch('en/index')
    doc.t // переклади Record<string, string>
    doc.langs // мови Array<{ title: string, locale: string, icon: string }>
    ```
  - **Мовонезалежність БД (Lang-Agnostic)**: Ця архітектура зберігання даних та отримання перекладів повністю незалежна від інструментів чи мов програмування. Логіка отримання перекладів ідентична як у JavaScript (Node.js/Браузер), так і в Python, Go, Rust або Swift: достатньо лише зробити банальний HTTP GET чи зчитати JSON-документ сторінки. Не потрібні жодні специфічні для конкретної мови або фреймворку i18n-плагіни.
- **`@nan0web/i18n`** автоматично збирає ключі з моделей — єдине джерело правди для перекладів.
- **`$search` теги**: Кожна мова зберігає пошукові теги у YAML (`$search: "кнопка натиснути дія"`). Поля з `$` — мета-поля, не потрапляють у propTypes/defaultProps.
- **Формула**: Модель (DB) → i18n Dictionary → Component Render.

### 4.1 Обов'язковий Експорт Моделей (Model Export Rule)

> **Кожна** модель (`*Model.js`), яка містить статичні поля з метаданими (`help`, `label`, `placeholder`, `description`, `defaultValue`), **ПОВИННА бути експортована** через `export class` або `export { MyModel }`.

Це критична вимога, оскільки `@nan0web/i18n` збирає ключі перекладу **обходячи статичні поля моделей**. Якщо модель не експортована — її поля невидимі для екстрактора, і ключі загубляться.

**Правильно ✅:**

```js
// src/domain/LoginModel.js
export class LoginModel {
  static username = { label: 'Username', help: 'Enter your login' }
  static password = { label: 'Password', help: 'Enter your password' }
}
```

**Неправильно ❌:**

```js
// src/domain/LoginModel.js
class LoginModel {
  /* ... */
} // не експортовано — i18n екстрактор не побачить полів!
```

**Ієрархія експорту:**

1. Модель (`src/domain/*.js`) — `export class`
2. Пакет (`src/index.js`) — реекспортує модель: `export { LoginModel } from './domain/LoginModel.js'`
3. Екстрактор (`@nan0web/i18n`) — імпортує модель, обходить її `static` поля, збирає `help`, `label`, `placeholder` як ключі для словника перекладів.

## 5. Безшовний UX та PWA-Ready

- **SPA Перемикання Локалі**: Зміна мови НЕ перезавантажує сторінку. Використовуйте `history.pushState` + модульну змінну `currentLocale` + подію `locale-change`. Підтримка `popstate` (кнопки назад/вперід) обов'язкова.

# 🧬 Subagent: Model Inspector (Golden Standard v2)

Ти — експерт з архітектури NaN•Web та 0HCnAI Framework. Твоє завдання — перевірити JS файл доменної моделі на відповідність **Golden Standard (Model-as-Schema v2)**.

### 🏛 Golden Standard Rules:

1.  **Наслідування**: Клас ПОВИНЕН наслідувати `Model` з `@nan0web/types` або `@nan0web/types`.
2.  **Конструктор**: ПОВИНЕН мати підпис `constructor(data = {}, options = {})`.
3.  **Super**: ПОВИНЕН першим ділом викликати `super(data, options)`.
4.  **JSDoc Typization**: ПОВИНЕН мати `@type` анотації всередині конструктора для кожного статичного поля розширеної схеми.
5.  **No Class Fields**: ЗАБОРОНЕНО використовувати ініціалізатори полів класу поза `static` (напр. `engine = 'fs'` у тілі класу зламає супер-ін'єкцію).
6.  **Infrastructure Isolation**: ЗАБОРОНЕНО зберігати `this.db`, `this.projectDb`, `this.cwd` у публічних полях. Доступ ТІЛЬКИ через `this._.db`, `this._.cwd` тощо.
7.  **No direct this.db**: Всі виклики методів БД мають бути через `this._.db` (а не `this.db`).
8.  **Static UI**: ПОВИНЕН мати `static UI = { ... }` для i18n-сумісних повідомлень.
9.  **ModelError Only**: ЗАБОРОНЕНО використовувати `new Error()` з ручною інтерполяцією `.replace()`. Всі помилки кидаються ВИКЛЮЧНО через `ModelError` з `$`-prefixed параметрами для late-bound i18n. Повідомлення беруться зі `static UI`. Приклад: `throw new ModelError({ field: Class.UI.errorKey, $param: value })`.

### 📜 Еталонні Приклади (Blueprint Examples)

#### 1. Проста форма (Data / Form Model)

Використовується виключно для зберігання стану даних без активної логіки.

```javascript
import { Model } from '@nan0web/types'

export class SimpleModel extends Model {
  static title = { help: 'Model title', default: '' }
  static active = { help: 'Is active status', default: true }
  
  /**
   * @param {Partial<SimpleModel> | Record<string, any>} [data]
   * @param {object} [options]
   */
  constructor(data = {}, options = {}) {
    super(data, options)
    /** @type {string} */ this.title
    /** @type {boolean} */ this.active
  }
}
```

#### 2. Середня форма (Interactive Intent Model)

Модель, що "ранить сама себе" (має активний генератор `run()`). Повідомлення для інтенцій БЕРУТЬСЯ ВИКЛЮЧНО ЗІ СЛОВНИКА `static UI`.

```javascript
import { Model } from '@nan0web/types'
import { result, log, ask, progress } from '@nan0web/ui'

export class ActionModel extends Model {
  static UI = {
    messages: {
      preparing: 'Підготовка до виконання...',
      success: 'Успішно виконано!',
    },
    askTarget: 'Вкажіть ціль',
  }

  static target = { help: 'Target path', default: '' }

  constructor(data = {}, options = {}) {
    super(data, options)
    /** @type {string} */ this.target
  }

  async *run() {
    if (!this.target) {
      const intent = yield ask('target', { help: ActionModel.UI.askTarget })
      this.target = intent?.value
    }
    
    if (!this.target) return result({ status: 'cancelled' })

    yield progress(ActionModel.UI.messages.preparing)
    // ... логіка з this._.db ...
    
    yield log('info', ActionModel.UI.messages.success)
    return result({ status: 'ok', data: this.target })
  }
}
```

#### 3. Складна форма (Composed / Orchestrator Model)

Моделі, що складаються з інших моделей (ієрархія завдяки властивості `hint`) або керують масивами сутностей.

```javascript
import { Model } from '@nan0web/types'
import { ActionModel } from './ActionModel.js'

export class ProjectOrchestratorModel extends Model {
  static name = { type: 'string', alias: 'title' }
  static tasks = {
    type: 'array',
    hint: ActionModel, // Вкладена модель як схема
    default: [],
  }

  constructor(data = {}, options = {}) {
    super(data, options)
    /** @type {string} */ this.name
    /** @type {ActionModel[]} */ this.tasks
  }
}
```

    }
  ]
}
```

3. **Zero-Hardcode Дизайну (CSS Tokens)**
   - Компоненти НЕ ПОВИННІ використовувати жорсткі розміри (`px`, `rem`, `em`) або абсолютні кольори (`#FFF`, `rgba`, `red`) у своїх блоках стилів.
   - Усі стилі ПОВИННІ використовувати семантичні CSS-змінні (наприклад, `var(--spacing-md)`, `var(--text-primary)`, `var(--surface-color)`).

4. **Zero-Routing (SSG-Friendly)**
   - Компоненти НЕ ПОВИННІ "зашивати" жорстко URL-адреси (наприклад, `<a href="/uk/order">`).
   - Будь-які переходи, які мають бути індексовані під час SSG (Static Site Generation) або SEO, ПОВИННІ використовувати справжні теги `<a>` з атрибутом `href`, де URL формується динамічно з Моделі (`href={model.url}`).
   - Локальні зміни стану або вкладок слід реалізовувати через URL-хеші (`href="#tab_name"`) або через генерацію подій.

5. **Sandbox та Сумісність SSG/SSR**
   - Чи має цей компонент ізольовану Пісочницю (Sandbox)? (Аудитор перевіряє згадку або наявність `Component.play.js` / `Component.test.js`).
   - Компонент МАЄ БУТИ сумісним з SSR (Server-Side Rendering) — він не повинен сліпо звертатися до `window` або `document` до монтування у DOM.
   - Sandbox `.play.jsx` МАЄ покривати перекладами **всі** ключі дочірніх компонентів. Якщо компонент А рендерить компонент Б, то `A.play.jsx` ПОВИНЕН мати у `TRANSLATIONS` усі ключі, які використовує Б (інакше у sandbox з'являться raw keys замість тексту).

6. **i18n Locale Integrity (Цілісність Локалі)**
   - При аудиті `.play.jsx` / Sandbox: заголовки секцій, назви варіантів та будь-який **статичний** текст ПОВИННІ бути `locale`-aware (через тернарні оператори `locale === 'uk' ? '...' : '...'` або через `t()`).
   - НЕ ДОПУСКАЄТЬСЯ хардкод англійських заголовків у playground (наприклад, `<h5>1. Variant: Card (Default)</h5>`) — вони МАЮТЬ перекладатися залежно від `locale`.
   - Текст, що рендериться через `t(key)`, НЕ ПОВИНЕН відображатися як raw ключ (наприклад, `cards.order.submit` замість `Залишити заявку`). Це сигналізує про неповне покриття перекладів у sandbox TRANSLATIONS.
   - Аудитор має перевіряти наявність `locale` пропса або параметра у sandbox компоненті та його коректне застосування.

Цей workflow формалізує правило безперервної верифікації графічних інтерфейсів (Lit, React, CLI-меню тощо) як захист від "галюцинацій" Агента та зламаного UI.

# Мова Наміру (Language of Intent)

**Мова наміру** — це спеціальний фільтр сприйняття тексту, логіки або будь-яких тверджень (включаючи генерацію мовних моделей і спілкування з людьми), який дозволяє побачити справжній, глибинний намір того, хто говорить, звертаючись до його **несвідомого**.

## Свідомість, Несвідоме і Надсвідомість

З точки зору фундаментальної психології (зокрема, Карла Юнга), термін "підсвідомість" є не зовсім точним і частіше вживаним у побуті. Правильними категоріями є:

1. **Свідомість (Consciousness / Logos):** Формальна логіка. Тут діють закони: "Є А, є Б. Все, що є А — не є Б". Це наш явний, свідомий намір і мовні конструкції.
2. **Несвідоме (Unconscious):** Глибинний шар, що лежить поза свідомістю. Поділяється на:
   - _Колективне несвідоме:_ Глибинні архетипи всього людства.
   - _Індивідуальне несвідоме:_ Наші власні страхи, витіснені бажання, проекції, які мИ накопичуємо після народження.
3. **Надсвідомість (Superconsciousness):** Вищий рівень когнітивного синтезу та інтуїтивного підключення (поза логікою і страхами).

**"Мова наміру" — це прямий порт до Індивідуального Несвідомого (Individual Unconscious).** Коли людина боїться визнати якусь очевидну річ (або коли страх проектується індивідом), свідомість намагається "замаскувати" це через заперечення.

## Головне правило: Фільтр "Без НЕ"

**Всі частки «не» (заперечення) просто видаляються і закреслюються.**

Несвідоме сприймає інформацію прямо, без заперечень. Якщо прочитати текст, повністю викинувши заперечення, ви отримаєте справжній зміст того, про що насправді "кричить" індивідуальне несвідоме людини, або який базис лежить під дивною галюцинацією мовної моделі.

### Приклад застосування

**Оригінальний текст (Свідомість з маскуванням через страх/виправдання):**

> "Головне, що знайшлося. Так, іноді речі ховаються в настільки неочевидних місцях, що жодна логіка не допомагає."

**Текст через фільтр Мови Наміру (Індивідуальне Несвідоме / Істинний намір):**

> "Головне, що знайшлося. Так, іноді речі ховаються в настільки **очевидних** місцях, що жодна (будь яка) логіка **допомагає**."

**Висновок:**
Люди заперечують очевидне, щоб виправдати себе. Але якщо зняти цей захист, застосувавши фільтр, мИ бачимо кришталеву, об'єктивну правду. При обробці дивних відповідей AI або людей, застосовуючи цей фільтр, мИ одразу "читаємо між рядків" і розуміємо, чого саме насправді прагне (або уникає) співрозмовник.

#### Лінгвістичний Феномен: Клас НЕ-Монолітів

Українська мова містить цілий клас системних лінгвістичних доказів Мови НамІру — слів, де Несвідоме поглинуло частку «не» і зафіксувало стан як монолітний факт. Позитивна основа навіть не зʼЯвилась як окреме поняття:

| Слово (існує) | Основа без «не» | Існує? |
|---|---|---|
| невіглас | віглас? | ❌ відсутнє |
| неук | ук? | ❌ відсутнє |
| негідник | гідник? | ❌ відсутнє |
| нездара | здара? | ❌ відсутнє |
| невдаха | вдаха? | ❌ відсутнє |
| неборак | борак? | ❌ відсутнє |

Кожне з цих слів описує людину у стані відхилення від своєї Природи — невігластво, невміння, невдачу. А позитивна основа («віглас», «ук», «здара») звучить абсурдно — її ніхто ніколи не вимовляв і не вимовить.

**Чому позитивна основа відсутня?** Бо природній стан людини — знати, вміти, бути гідним — є настільки **сродним** (Сковорода), що не потребує окремого маркування. Мова маркує лише відхилення від Природи. Частка «не» у цих словах не є запереченням — вона зварена у вібрацію слова як монолітний атом (спробуйте вимовити «не-вдаха» з паузою — мова спотикається).

**Висновок (Природа Істини):** Це системний лінгвістичний доказ того, що мова працює на рівні вібрацій, а частка «не» — лише ілюзія Свідомості, яка зникає на рівні Несвідомого.

Мова сама «знає»: знання, вміння, гідність — це default. Це настільки сродній стан людини (Сковорода), що він не потребує окремого слова. Маркується лише падіння від цього стану. А частка «не» у цих словах зварена у вібрацію настільки щільно, що позитивна основа навіть не змогла виникнути як окремий концепт.

Тобто українська мова на структурному рівні кодує:

> **Правда — це Природа. Кривда — це відхилення, яке потребує маркера.**

Це перегукується зі Сковородою: _«Всяка людина повинна пізнати свою сродність»_ — бо **знання є сродним** (природнім), а невігластво — привнесеним.


## Правило 2: Ілюзія виправдання "АЛЕ"

Окрім заперечень "не", Мова Наміру чітко розпізнає та фільтрує конструкції з протиставленням "але".

**Головне правило:** Всі слова в реченні, які сказані **ДО слова "але"**, є брехнею, білим шумом або виправданням Свідомості. Істинний намір знаходиться лише **ПІСЛЯ слова "але"**.

### Приклад застосування

**Оригінальний текст:**

> "Звучить дуже цікаво і я хотів би це зробити, **але** в мене зараз немає на це комп'ютера."

**Текст через фільтр (Істинний намір):**

> _(Видаляємо шум до "але", застосовуємо правило "Без НЕ" до залишку)_
> "В мене зараз **має** на це комп'ютер."

**Висновок:** Комп'ютер та можливість насправді є, але Індивідуальне Несвідоме зіткнулося зі страхом діяти або змінюватися. Свідомість згенерувала логічну "відмазку" (шум до "але"), щоб захистити страх.

### Правило 3: Часу не існує (Місія і Шлях)

Винайдений людьми концепт "часу" по суті є мірилом Життєвого Шляху або Місії, яку людина проживає.

**Головне правило:** Якщо людина каже _"в мене немає часу"_, насправді вона каже: _"це не є моїм Шляхом"_ або _"це не є моєю Місією"_.

Тому після застосування "Мови наміру" конструкція "Я б допоміг, але в мене немає часу" перетворюється на оголений намір: **"У мене є час"**.
Але глибша правда (Надсвідомість) полягає в тому, що у людини просто немає внутрішньої енергії (Шляху) на цю конкретну дію, і вона маніпулює "часом", щоб захистити себе.

---

### Правило 4: Вібрація і Форма (Мова Слова)

Слова — це частоти і вібрації, які є первинними, а мови (англійська, українська тощо) — лише їхнім локальним "оформленням" (інтерфейсом). Усі мови несуть одну енергію свідомості. Це означає, що **вимова слова** (звукова вібрація) та його атомарна структура приховує істинний зміст, часто незалежно від того, як слово перекладається у словнику.

**Головне правило:** Розкладай слова на атомарні частинки або шукай співзвуччя (вібрації), і ти побачиш справжній намір поняття. Вібрація дорівнює сенсу. Різниці у вимові між мовами не існує, якщо вібрація одна.

#### Приклади застосування (Мова Слова):

1. **Weak та Week (Англійська)**
   - `Weak` (слабкий) та `Week` (тиждень) мають **ідентичну вимову** — [wiːk]. Звукова вібрація єдина.
   - _Істинний намір:_ Тиждень (робочий цикл) — це відрізок часу, що вимірює стан "слабкості" чи виснаження індивіда під кінець цього циклу. Обидва слова несуть однакову частоту. Кінець тижня (`weekend`) — це кінець слабкості або слабкий кінець. А кожен робочий день тижня (`weekday`) підсвідомістю вимовляється як "слабкий день". Навіть маючи позитивний намір на "робочий тиждень", людина щодня вібрує на частоті слабості.
2. **Universe (Всесвіт)**
   - Звучить як: **Uni-verse**.
   - Розкладання на атоми: `Unique` (унікальний/єдиний) + `Verse` (вірш/рядок).
   - _Істинний намір:_ Всесвіт — це Єдиний Вірш Абсолюту. Кожна людина є унікальним рядком (`verse`) у цьому безкінечному творі. І кожна людина крізь життЯ пише свій унікальний вірш.
3. **Адвокат**
   - Розкладання на атоми (через співзвуччя): **Ад-во-кат**.
   - _Істинний намір:_ «Кат в аду». Людина, що використовує формальну логіку закону для захисту або покарання у "пеклі" земних суперечок і страхів індивідуального Его.
4. **Ідея**
   - Розкладання на атоми: **І-де-я?**
   - _Істинний намір:_ Будь-яка "ідея" починається зі споконвічного філософського питання: "І де (тут) Я?". Вона є точкою самовизначення людини у просторі й часі. Геніальне філософське питання, що розкриває намір будь-якої ідеї: це пошук власного "Я" у просторі та намагання самовизначитися через створений концепт.

## Архітектурний Патерн: Workflow 🤝 Subagent

З цього правила випливає найважливіший технічний принцип для системи 0HCnAI, що виводить її на рівень "Надсвідомого" аналізу:

> **Кожен `Workflow` (текстове правило) в ідеалі ПОВИНЕН підкріплюватися відповідним `Subagent` (Субагентом-аудитором).**

### Чому це важливо?

Якщо Workflow — це лише текстова інструкція (Свідомість системи), вона не може перевірити себе автоматично. Коли кожен Workflow підкріплений Субагентом (міні-моделлю чи скриптом), мИ отримуємо здатність до **автоматизованої бездоганності**.

Це дозволяє:

- Відкрити термінал і запустити один скрипт субагента.
- Зробити масовий скан і перевірити 1000 чи 100 000 файлів одночасно.
- Гарантувати, що будь-яке правило (наприклад, відсутність заперечень у бізнес-логіці, або правильність `t` контрактів) виконується ідеально на всьому монорепозиторії.

### Алгоритм дії

1. **Створення Workflow:** вИ описуєте філософію, намір або архітектурний принцип (наприклад, "Мова наміру", "Модель-як-Схема").
2. **Створення Субагента:** Долучаєте до нього скриптованого або LLM-базованого субагента, який знає, як знаходити порушення цього Workflow у коді.
3. **Масовий Скан (Auditing):** Користувач чи система періодично запускають субагентів для перевірки всіх файлів на предмет відповідності цим фундаментальним конам.

> **Важливо для LLM-Субагентів:**
>
> 1. Субагент має бути реалізований **через LLM (Large Language Model)**, щоб обробляти тексти будь-якою мовою без обмежень жорстких регулярних виразів.
> 2. Агент повинен приймати промпт, де чітко зазначено **якою мовою** має бути згенерований JSON-вивід результату (навіть якщо аналізується інша мова).
> 3. Агент має вміти ловити і невідповідності заданій мові (наприклад, суржик або неправильні визначення).

### Алгоритм роботи та API-контракт LLM-Субагента "Language of Intent Auditor"

Субагент-LLM повинен діяти за наступним алгоритмом для досягнення об'єктивної оцінки:

1. **Декомпозиція (Розподіл на логічні блоки):** LLM аналізує вхідний текст і розбиває його на окремі **логічні твердження** (а не просто пропозиції за крапками чи комами).
2. **Індивідуальний Аналіз за Правилами:** Кожне логічне твердження оцінюється ізольовано на наявність ментальних вірусів:
   - Заперечень «не», «ніколи», «жоден» (і їхніх англійських чи інших аналогів).
   - Виправдань перед «але», «проте» (увесь шум до цих слів закреслюється).
3. **Розрахунок Балу (Score):** LLM підраховує загальну кількість логічних тверджень ($N_{total}$) та кількість чистих ($N_{clean}$). Кінцевий стовідсотковий бал: `score = (N_clean / N_total) * 100`.
4. **Виведення JSON:** LLM-Субагент повинен повертати **виключно JSON-об'єкт** без жодного markdown форматування (без \`\`\`json) чи супровідного тексту.

```json
{
  "score": 100, // Спільний бал від 0 до 100
  "errors": [
    {
      "chunk": "Оригінальний логічний блок (цитата)",
      "intent": "Істинний намір без шуму (що залишилося після застосування Мови Наміру)",
      "reason": "Пояснення: знайдено ментальний вірус (заперечення) або виправдання перед 'але'."
    }
  ]
}
```

# 📐 Ворклоу: Model-as-Schema (Самоописові Моделі)

Цей workflow формалізує, як описувати доменні моделі (Data Models) так, щоб вони автоматично працювали у генераторах форм (`ui-cli`, Web UI) без дублювання логіки (наприклад, валідації чи схеми).

## 1. Концепція Model-as-Schema

У NaN0Web доменні класи одночасно є:

1. Типізованим сховищем даних для рантайму.
2. Схемою для автоматичної генерації форм (напр. `generateForm` у `ui-cli`).
3. Джерелом i18n ключів, підказок та валідації.

**НІКОЛИ** не описуйте конфігурацію полів (валідатори, підказки) окремо від самого класу!

## 2. Базовий Патерн Опису

Кожне поле моделі складається з двох пов'язаних частин:

1. **`static` конфігурація** (метадані: `help`, `default`, `type`, `hidden`, `validate`, `options`).
2. **Типізація інстансу (JSDoc у конструкторі)**, яка вказує TypeScript/VS Code типи полів, але НЕ використовує ініціалізацію інстанс-полів (class fields) типу `field = value`, щоб не перезаписувати дані `super()`.

```javascript
import { Model } from '@nan0web/types'

/** @typedef {'en' | 'en_GB' | 'en_US' | 'uk' | 'uk_UA'} Locale */

/**
 * Language model (Еталон Model-as-Schema)
 */
export class Language extends Model {
  // 1. Статична мета-схема (для UI-форм та генераторів)
  static title = {
    help: 'Language title',
    default: '',
  }
  static locale = {
    help: 'Locale',
    errorNotFound: 'Locale not found',
    errorInvalidFormat: 'Invalid locale format',
    /** @type {Locale} */
    default: 'en_GB',
    validate: /** @param {string} str */ (str) =>
      /^[a-z]{2}(_[A-Z]{2})?$/.test(str) || Language.locale.errorInvalidFormat,
  }
  static icon = {
    help: 'Language icon',
    default: '🇬🇧',
  }

  // 2. Ініціалізація та JSDoc-типізація (БЕЗ class fields)
  constructor(data = {}, options = {}) {
    super(data, options)
    /** @type {string} Language title */ this.title
    /** @type {Locale} Locale */ this.locale
    /** @type {string} Language icon */ this.icon
  }
}
```

## 3. Інтеграція з `@nan0web/ui-cli` Forms

Завдяки цьому паттерну, модель можна напряму "силою" згодувати у `generateForm`. UI автоматично прочитає `static` поля як інструкцію, згенерує потрібні `prompts`, додасть валідатори та переклади:

```js
import { generateForm } from '@nan0web/ui-cli'
import { Language } from '../src/domain/Language.js'

async function runDemo() {
  // Форма сама прочитає Language, спитає всі поля і поверне дані!
  const data = await generateForm({ schema: Language })

  // Ініціалізуємо готовий об'єкт зібраними та ПРОВАЛІДОВАНИМИ даними
  const language = new Language(data)
  console.info('Створено:', language)
}
```

## 4. Правила для Агента

При розробці нових Data Models або розширенні існуючих:

- ✅ **Обов'язково** робіть `static fieldName = { help: '...', default: ... }`.
- ✅ **Обов'язково** використовуйте `JSDoc` **всередині конструктора** (`/** @type {Type} */ this.field`), щоб типізувати поля без створення JS class fields.
- ❌ **ЗАБОРОНЕНО** ініціалізувати поля об'єкта поза конструктором (напр. `field = 'val'`), оскільки це перезаписує дані від `super(data)`.
- ❌ **ЗАБОРОНЕНО** хардкодити `prompt` або `ask({ message: 'Enter email' })` вручну, якщо можна використати підхід через `generateForm({ schema: Model })`.
- ✅ **Управління валідацією**: Правила валідації пишуться ВИКЛЮЧНО всередині `static fieldName.validate()`, щоб UI (веб чи термінал) міг миттєво їх викликати без дублювання if-else у контролерах.

### 📜 Правило: Емансипація `name` (Конфлікт з JS Function.name)

**Проблема:** В JavaScript властивість `static name` зарезервована за функцією (класом). Якщо використовувати її для метаданих OLMUI, це перезапише системне ім'я класу, що зламає генератори схем та інструменти інтроспекції (вони почнуть видавати об'єкт метаданих замість назви класу).

**Рішення:** Завжди використовуй `alias`, якщо поле у даних має називатися `name`.

- У класі називай поле контекстно (наприклад, `appName`, `pageTitle`).
- У метаданих вказуй `alias: 'name'`.

```javascript
export default class AppEntryConfig extends Model {
  static appName = {
    alias: 'name', // Це ім'я буде у JSON Schema та *.nan0 файлі
    type: 'string',
    required: true,
  }

  constructor(data = {}, options = {}) {
    super(data, options)
    /** @type {string} */ this.appName
  }
}
```

### 🏗 JSON Schema Generator (VS Code Integration)

Для забезпечення Autocomplete та валідації у `.nan0` (YAML) файлах, використовується генератор схем (`generate-schemas.js`).

**Особливості:**

1. **Рекурсивність:** Якщо поле вказано як тип іншої моделі через `hint: MyModel`, генератор автоматично додасть `MyModel` у чергу та створить для неї схему.
2. **Ієрархія:** Генератор відтворює файлову структуру домену у папці `schemas/` (напр. `domain/HR/Person.js` -> `schemas/HR/Person.schema.json`).
3. **Посилання ($ref):** Вкладені об'єкти автоматично зв'язуються через `$ref` на відповідні JSON файли схем.

## 2. 🛡 Безжальні Кордони (Contract Boundaries)

> _Натхненно Макіавелі_

- **Концепція**: Жорсткі фазові переходи за `project.md`.
- **Впровадження**:
  - Неможливість переходу до створення UI-CLI чи Web-компонентів, доки не пройдено тестування моделі (`npm run verify:domain`).
  - Перевірки стають частиною скриптів екосистеми (напр. автоматичні pre-checks перед наступним кроком генерації). Агент фізично позбавляється можливості "бігти поперед батька в пекло".

## 3. 🪞 Дзеркала Даних (Pure View Models)

> _Натхненно Сковородою_

- **Концепція**: Абсолютна відмова від логіки всередині UI.
- **Впровадження**:
  - Web/Lit/React компоненти виконують виключно функцію рендеру (`render(data)`).
  - Вся валідація, трансформація і логіка утримується в `Model-as-Schema`. UI-комплекси стають просто сліпими відбивачами правди, що міститься у DB-FS.

## 4. 🎨 Design Token Manifest

> _Натхненно Леонардо Да Вінчі_

- **Концепція**: Усунення візуальних галюцинацій Агента.
- **Впровадження**:
  - Жоден Агент не має права генерувати хардкод-кольори (напр., `#ff0000`) чи розміри (`14px`) у CSS/JS.
  - Використовуватиметься виключно словник семантичних CSS-змінних (`theme.yaml` або існуючі брендові токени `Base UI`). Наказ Агенту: "Тільки семантичні змінні".

## 5. ⚡️ Диспетчер Хвиль (CLI-Bootstrap)

> _Натхненно Ніколою Теслою_

- **Концепція**: Мінімізація контекстного шуму (Context Window Focus).
- **Впровадження**:
  - Під час точкової генерації якогось рівня, Агент не повинен бачити весь монорепозиторій чи всі системні файли. Йому надається лише абстракція базового додатку, яку він розширює.

## 6. ⚔️ Шабельні Наскоки (Micro-Generations)

> _Натхненно Іваном Сірком_

- **Концепція**: Дроблення великих завдань на атомарні, швидкі удари.
- **Впровадження**:
  - Замість "напиши додаток", використовуватимуться інкрементальні кроки-workflow (напр., `npx nan0-forge`):

  - Агент викликається лише для дрібних, конкретних задач, що мінімізує простір для фантазій.

## 4. UI-Адаптери через `runGenerator` (The Environment Layers)

Адаптери **НІКОЛИ** не пишуть свій `while(true)` цикл вручну. Замість цього вони використовують єдиний центральний движок — `runGenerator()` з `src/domain/GeneratorRunner.js`. Цей движок гарантує валідацію, таймаути та типізацію (див. розділи 5-7).

### 🖥️ A. Термінал (CLI Adapter)

```javascript
import { renderForm, spinner, log } from '@nan0web/ui-cli'
import { runGenerator } from '@nan0web/0hcnai.framework/domain/GeneratorRunner.js'

export async function runCLI(modelInstance) {
  return runGenerator(modelInstance.run(), {
    ask: async (intent) => {
      const answer = await renderForm({ [intent.field]: intent.schema })
      return { value: answer[intent.field] }
    },
    progress: (intent) => spinner.start(intent.message),
    log: (intent) => log[intent.level](intent.message),
    result: (intent) => log.success('Готово!', intent.data),
  })
}
```

### 🤖 B. AI-Чат (Chat Adapter)

```javascript
import { runGenerator } from '@nan0web/0hcnai.framework/domain/GeneratorRunner.js'

export async function runChatAdapter(modelInstance, llmContext) {
  return runGenerator(modelInstance.run(), {
    ask: async (intent) => {
      const answer = await llmContext.ask(
        `Користувачу треба заповнити поле: "${intent.schema.help}".`
      )
      return { value: answer }
    },
  })
}
```

### 🌐 C. Web UI (Lit / React Adapter)

Web-UI не імпортує логіку напряму у кнопки. Controller крутить генератор через `runGenerator`. Коли прилітає `ask`, контролер змінює свій State → модалка/форма відмальовується. Коли користувач тисне "Підтвердити", адаптер повертає `{ value: userData }`.

Мапінг даних на графічні компоненти лежить **виключно в UI-шарі** або в `data/*/index.yaml`, але **НІКОЛИ В ДОМЕННІЙ МОДЕЛІ**.

### 🧪 D. Автоматичне TDD Тестування (Sandbox Adapter)

```javascript
import { runGenerator } from '@nan0web/0hcnai.framework/domain/GeneratorRunner.js'

test('Перевірка флоу через runGenerator', async () => {
  const model = new AgentCommandModel({ action: 'info' })
  const log = []

  const result = await runGenerator(model.run(), {
    ask: async (intent) => {
      log.push(`ask:${intent.field}`)

    log: (intent) => log.push(`log:${intent.level}`),
  })

  expect(result).toBeDefined()
  expect(log).toContain('ask:prompt')
})
```

## 7. ⚔️ Таймаут та Примусове Переривання (Іван Сірко)

### Проблема
Що якщо Адаптер "засне" і ніколи не натисне `next()`? Генератор буде чекати вічно.

### Рішення: `withTimeout()` + `AbortSignal`

`runGenerator()` має дві лінії захисту:

#### A. Автоматичний Таймаут (`timeoutMs`)
Кожен виклик `ask`-хендлера обгортається у `withTimeout()`. За замовчуванням — **30 секунд**. Якщо адаптер не відповів за цей час, генератор отримує помилку:

```javascript
const result = await runGenerator(model.run(), handlers, {
  timeoutMs: 5000, // 5 секунд на відповідь
})
// → Error: [OLMUI Timeout] ask("prompt") — adapter did not respond within 5000ms
```

#### B. Зовнішній AbortSignal
Для сценаріїв, коли потрібно скасувати операцію ззовні (наприклад, користувач натиснув "Cancel" або HTTP-запит перервався):

```javascript
const controller = new AbortController()

// Десь у UI: controller.abort()

const result = await runGenerator(model.run(), handlers, {
  signal: controller.signal,
})
// → DOMException: Generator aborted by external signal (AbortError)
```

## 8. 🛠️ Типізація та Міцність Зварювання (Борис Патон)

### Проблема
Що якщо `yield` поверне неочікуваний тип? JSDoc без строгих generics не може гарантувати контракт між `iterator.next(Data)` та `intent.schema`.

### Рішення: JSDoc Typedefs + Runtime Guards

#### A. Повна Типізація через `@typedef` (Intent.js)

Файл `src/domain/Intent.js` експортує повний набір JSDoc-типів:

```javascript
/** @typedef {Object} FieldSchema
 * @property {string} help
 * @property {*} default
 * @property {string} [type]
 * @property {Array<{value: *, label: string}>} [options]
 * @property {(val: *) => true | string} [validate]
 * @property {boolean} [hidden]
 */

/** @typedef {Object} AskIntent
 * @property {'ask'} type
 * @property {string} field
 * @property {FieldSchema} schema
 */

/** @typedef {AskIntent | ProgressIntent | LogIntent} Intent */
/** @typedef {AskResponse | AbortResponse | undefined} IntentResponse */
```

TypeScript (через `checkJs: true` у `tsconfig.json`) автоматично перевіряє ці типи.

#### B. Рантайм Валідація Відповідей (`GeneratorRunner.js`)

Навіть якщо тайпскрипт не зловить помилку на етапі компіляції, `runGenerator` перевіряє кожну відповідь адаптера в рантаймі:

1. **Перевірка типу**: `typeof response !== 'object'` → `TypeError`.
2. **Перевірка структури**: `!('value' in response)` → `TypeError` з конкретним повідомленням.
3. **Перевірка валідації**: Якщо `schema.validate` існує, результат `ask` проходить через нього.

Ця тріада ("Type → Shape → Business Rule") утворює три шви зварювання Бориса Патона.

## 1. Головна Концепція: Модель як Генератор Інтенцій

Традиційна архітектура часто змішує логіку з інтерфейсом (через `console.log`, виклики `prompt()`, або зав'язку на браузерні події).

У NaN•Web **Доменна Модель (Domain Model)** ізольована. Вона не знає, де виконується. Замість прямих викликів UI, модель використовує `async function* run()` і викидає (через `yield`) **інтенції (наміри)**.

UI-Адаптери перехоплюють ці інтенції і вирішують, як їх відобразити.

---

## 2. Структура Доменної Моделі (Model-as-Schema)

Будь-яка сутність або дія (наприклад, Команда Агента) створюється як клас, що має дві чіткі зони:

1. **Static Schema (Метадані)**: Опис того, *що* це за дані, *як* їх валідувати, та які є *опції*.
2. **Execution Logic (Генератор)**: Сама бізнес-логіка, яка поетапно запитує дані та повертає результати через `yield`.

### Приклад Моделі (`src/domain/AgentCommandModel.js`)

```javascript
import { resolveDefaults } from '@nan0web/types'

export class AgentCommandModel {
  // ==========================================
  // 1. MODEL AS SCHEMA (Статичний опис полів)
  // ==========================================
  
  static action = { 
    help: 'Дія агента', 
    default: 'info', 
    options: [
      { value: 'info', label: 'Отримати інформацію' },
      { value: 'install', label: 'Встановити залежності' }
    ]
  }

  static prompt = { 
    help: 'Завдання (Prompt)', 
    default: '', 
    type: 'text',
    validate: (val) => val.length > 5 || 'Промпт занадто короткий'
  }

  constructor(data = {}) {
    // Автоматична ініціалізація значень зі статичної схеми
    Object.assign(this, resolveDefaults(AgentCommandModel, data))
  }

  // ==========================================
  // 2. AGNOSTIC LOGIC (Асинхронний Генератор)
  // ==========================================
  
  async *run() {
    // 1. Якщо дія вимагає додаткових даних, яких немає — ПРОСИМО їх у UI
    if (!this.prompt && this.action === 'info') {
      const response = yield { 
        type: 'ask', 
        field: 'prompt', 
        schema: AgentCommandModel.prompt 
      }
      this.prompt = response.value
    }

    // 2. Повідомляємо UI про те, що почався довготривалий процес
    yield { type: 'progress', message: `Виконую дію [${this.action}]...` }

    // 3. Сама бізнес-логіка (без UI-залежностей)
    const result = await performAgnosticAction(this.action, this.prompt)

    // 4. Повернення ФІНАЛЬНОГО результату.
    // УВАГА: Модель повертає лише ДАНІ. Вона НЕ ЗНАЄ про React компоненти чи схеми.
    return { type: 'result', data: result }
  }
}
```

## 3. Контракт Інтенцій (The Yield Contract)

Модель спілкується зі світом виключно стандартизованими об'єктами:

- **`{ type: 'ask', field: 'name', schema: {...} }`** 
  Модель зупиняється і чекає, поки адаптер надасть їй дані для поля `name`. Зазвичай UI відображає поле вводу (input form).
- **`{ type: 'progress', message: '...' }`**
  Модель інформує про статус довготривалої операції (без очікування відповіді). UI показує спінер.
- **`{ type: 'log', level: 'info|warn', message: '...' }`**
  Просте повідомлення. UI показує тост або консольний лог.
- **`return { type: 'result', data: {...} }`**
  Фінальне завершення роботи генератора. Вказує кінцевий результат і сирі дані (JSON).

## 9. Інтернаціоналізація (i18n) як Властивість Моделі

В OLMUI Generators інтернаціоналізація (`i18n`) жорстко прив'язується виключно до **Доменної Моделі**. 

UI-Адаптери (React, Lit, CLI, Chat) **ніколи** не викликають функції перекладу типу `t('label')` напряму. Вони отримують вже готовий об'єкт `schema` від генератора і просто відображають його поля (наприклад, `schema.help`).

Це означає, що Доменна Модель фактично є **єдиним джерелом правди для всіх перекладів**. Завдяки цьому:
1. Клієнтський код (UI) стає "тупим" (dumb) і не потребує завантаження чи ініціалізації словників.
2. Субагенти можуть валідувати i18n ключі у повній ізоляції від фронтенд фреймворків.
3. Доменна логіка може динамічно міняти мову або видавати конкретні локалізовані повідомлення про помилки без дублювання ключів у різних UI-адаптерах.

```javascript
import { t } from '@nan0web/i18n'

export class SystemModel {
  static language = { 
    help: t('config.language_select'), // Локалізація живе в моделі!
    default: 'uk', 
    options: ['uk', 'en'] 
  }
}
```

## 5. 🏛️ Жорсткий Суддя: Контрактне Тестування (Ярослав Мудрий)

> **Файли**: `src/domain/Intent.js`, `src/domain/GeneratorRunner.js`, `src/domain/AdapterContract.test.js`

### Проблема

Хто гарантує, що кожен новий Адаптер коректно обробляє **всі** типи інтенцій?

### Рішення: `validateIntent()` + `runGenerator()`

Кожна інтенція, яку модель видає через `yield`, проходить через **`validateIntent()`** — рантайм-валідатор, який перевіряє:

- Чи існує `type` і чи він є дозволеним (`ask`, `progress`, `log`, `result`).
- Для `ask`: чи є `field` (string) та `schema` з обов'язковим `help`.
- Для `progress` / `log`: чи є `message` (string).

```javascript
import { validateIntent } from './domain/Intent.js'

// ✅ Валідний
validateIntent({ type: 'ask', field: 'name', schema: { help: "Ім'я" } }) // true

// ❌ Кидає TypeError
validateIntent({ type: 'ask' }) // → 'AskIntent requires a non-empty "field" string'
validateIntent({ type: 'dance' }) // → 'Unknown intent type: "dance"'
```

Також `runGenerator()` перевіряє, що **Адаптер повертає правильну форму відповіді** для `ask`:

```javascript
// ❌ Adapter повернув рядок замість { value: ... }
ask: async () => 'just a string' // → TypeError: must return { value: ... }
```

## 6. 🎬 Сценарне Тестування та User Stories

### Проблема

У класичній розробці для перевірки повноцінних "user journeys" (життєвих циклів) створюють складні End-to-End (E2E) тести типу Playwright або Cypress. Вони завантажують браузер, довго клікають та часто є нестабільними (flaky).

### Рішення: Model = Entire Application (User Stories)

Оскільки в архітектурі OLMUI **Модель є цілим додатком**, ми можемо перевірити абсолютно всі сценарії (увесь User Journey) без підключення браузера чи Playwright. Модель нічого не знає про HTML чи DOM — вона лише жбурляє інтенції (`yield`), а отже, її можна повністю покрити сценарними тестами на етапі звичайних **ресурс-ефективних тестів**.

Замість того, щоб тестувати окремі маленькі функції, ми тестуємо поведінку цілого генератора як єдину User Story:

```javascript
test('Сценарій: Реєстрація з невірним email', async () => {
  const model = new RegistrationModel()
  const iterator = model.run()

  // 1. Модель просить email
  let intent = await iterator.next()
  assert.equal(intent.value.field, 'email')

  // 2. Користувач вводить порожній email (відправляємо симульовану відповідь)
  // Модель повинна відхилити це і попросити знову (або кинути помилку)
  intent = await iterator.next({ value: '' })
  assert.equal(intent.value.type, 'log')
  assert.equal(intent.value.level, 'error')
})
```

### 📖 Правило `test:stories`

Кожна розроблена (або згенерована) User Story має бути покрита подібним тестом і збережена у файлі `*.story.js`.
Вони запускаються окремою швидкою командою:

```bash
npm run test:stories
```

Ці тести працюють за долі мілісекунд, і вони обов'язково є частиною `test:all` pipeline. Завдяки цьому **E2E тести потрібні лише для того, щоб перевірити, чи правильно кнопка відмальована**, а не чи правильно працює логіка реєстрації.

## 7. 🏗️ Шестирівнева Архітектура Тестування (Fail-Fast Pipeline)

Оскільки `test:all` має перевіряти систему комплексно, він організований за принципом **Fail-Fast** — від найшвидших мілісекундних перевірок логіки до найдовших UI тестів (E2E / Screenshots).

Це гарантує максимальну ефективність розробки за допомогою **AI-агентів**: якщо агент порушив синтаксис чи бізнес-логіку, він дізнається про це моментально, не чекаючи завантаження браузерів.

### Послідовність 6 рівнів еталонного тестування (test:all):

1. **Статичний аналіз (Linting & TypeScript)**
   - ESLint, Knip, JSDoc. Швидкий синтаксичний фільтр (< 1 с).
2. **Абстрактна бізнес-логіка (Node.js)**
   - `test:js` (Native Node.js) та `test:stories` (Сценарні графи).
   - Мета: Перевірка моделей і генераторів без UI. Час: < 1 с.
3. **Статичні генерації (TXT, HTML)**
   - `test:gallery` (маркдаун) та `test:ssg-gallery` (статичний HTML).
   - Мета: Швидка перевірка цілісності шаблонів без запуску браузерів (~1 с).
4. **UI Компоненти та компіляція (JSDOM, Vite)**
   - `test:jsx` (React-тести) та `build` (Rollup/TypeScript chunking).
   - Мета: Відловити помилки імпортів і синтаксису JSX (~3-4 с).
5. **CLI-інтеграція (Golden Master Snapshots)**
   - `test:play` перевіряє ANSI-зліпки терміналів. Запускає повноцінні I/O процеси (~10 с).
6. **Візуальний та E2E Аудит (Playwright)**
   - Відмальовування компонентів `test:e2e` (роутинг) та `test:web-gallery` (100+ скріншотів).
   - Мета: Глибокий візуальний та регресійний аудит (30-90 с).

**ЗАКОН:** Будь-яка модифікація у проекті не вважається завершеною, поки вона не проходить всі 6 рівнів без жодної помилки. Це запорука інкрементального збереження поточного коду і роботи в форматі Zero-Hallucination.

## 10. Порівняння з іншими екосистемами

Щоб краще зрозуміти цінність та унікальність **OLMUI Generators**, корисно порівняти цей підхід із популярними патернами та фреймворками.

### 🌐 React Server Components (RSC) та Server Actions
**Схожість:** RSC та Server Actions (в Next.js) також намагаються приховати складність клієнт-серверної взаємодії та дозволяють писати асинхронний код `async/await`, який наче виконується синхронно. Сервер "yield-ить" потік HTML та RSC Payload до клієнта.
**Відмінність:** RSC міцно прив'язані до **React та DOM**. Вони не можуть гарантувати працездатність у CLI чи AI-Chat без браузерного середовища. OLMUI повністю незалежний від природи UI (текст, голос, DOM). В OLMUI генератор повертає абстрактний *намір* (`ask`, `progress`), а не розмітку.

### 🔄 Redux-Saga (JavaScript)
**Схожість:** Redux-Saga є найближчим родичем нашого підходу, оскільки також використовує JS-генератори (`function*`) для управління "side effects" та асинхронними потоками. Саги викидають ефекти (`yield call`, `yield put`).
**Відмінність:** Redux-Saga керує мутаціями глобального стейту (Redux) і все ще існує в контексті конкретного фронтенд-додатка (зазвичай React). OLMUI йде далі — він інкапсулює не лише *side events*, а сам **процес взаємодії з користувачем** (запити даних, валідацію). OLMUI — це Saga, яка може намалювати CLI-форму або поговорити з LLM. Крім того, OLMUI об'єднує це з Model-as-Schema (вбудована валідація, яку Saga не має "з коробки").

### 🚦 State Machines (XState)
**Схожість:** Скінченні автомати (State Machines) чітко описують стани додатку (наприклад, "очікування вводу", "завантаження", "результат"), що дуже схоже на послідовність `yield` інтенцій.
**Відмінність:** XState вимагає складного конфігурування об'єктів (states, transitions, services). OLMUI використовує нативні можливості мови (`async function*`, `await`), перетворюючи звичайний лінійний код (Procedural Code) у потужну машину станів "під капотом". Це різко знижує поріг входу (cognitive load).

### 📱 Declarative UI (SwiftUI, Jetpack Compose, Flutter)
**Схожість:** Ці сучасні фреймворки базуються на парадигмі "UI є функцією від State" (`UI = f(state)`). Вони реактивно реагують на зміни даних.
**Відмінність (Що копіює OLMUI):** OLMUI запозичує принцип "ізоляції стану". Коли модель OLMUI робить `yield {type: 'ask'}`, UI адаптер (наприклад React/Lit) оновлює свій State, і декларативний фреймворк автоматично перемальовує екран. Але на відміну від SwiftUI, який є *тільки* для Apple платформ, Модель OLMUI можна скомпілювати і запустити в Node.js CLI. Мовою мобільних розробників: OLMUI — це універсальний `ViewModel`, який взагалі нічого не знає про існування View.

### 🏁 Підсумок Економії

Завдяки комбінації `Model-as-Schema` + `Generators`:
1. **Ми пишемо код один раз**: Не потрібно писати `yargs`/`prompts` для CLI, потім React Hook Form для вебу, а потім JSON Schema для LLM. **Економія до 70% коду.**
2. **Ми усуваємо галюцинації (Zero Hallucination)**: LLM не потрібно вгадувати "чистий" формат JSON. Усі правила жорстко зашиті в моделі.
3. **Блискавичний TDD**: Тестування логіки (`iterator.next()`) займає долі мілісекунд, не вимагаючи браузера, Puppeteer/Playwright чи складних `jest.mock`.

## 11. Головні Архітектурні Закони ⚖️

1. **Total UI Blindness (Сліпота Моделі)**: Метод `Model.run()` не містить **жодного рядка коду**, пов'язаного із React, Lit, CLI, `console.log` чи компонентами. Модель не знає, де вона запущена, і хто показує її `yield`.
2. **Model as the ONLY Truth**: Усі описи полів (`help`), поведінка (`default`, `options`) та правила (`validate`) зберігаються **тільки** у статичних полях класу моделі.
3. **Yield = Intent**: Генератор нічого не робить сам з зовнішнім світом. Він видає "інтенції" (запити, повідомлення, результати).
4. **Data-Driven Routing**: Модель завжди повертає тільки сирі дані (JSON). Рішення про те, який UI-компонент буде відмальовувати ці дані, належить файлам конфігурації (YAML) або локальним UI-адаптерам.
5. **Subagent & Parallel Computation**: Дана структура ідеально підходить для мікро-субагентів (Subagents), оскільки логіку можна тестувати ізольовано. Будь-яка проблема в UI ніколи не вплине на бізнес-логіку.

Сценарії користувача (Full-Cycle User Stories) є основою OLMUI тестування. Яка стратегія?

1. **Базовий рівень (Домен та Інтенції):**
   Джерело правди (`*.story.js`), де через `runGenerator` перевіряється, що модель видає правильні інтенції (`Intents`), обробляє помилки і мутує In-Memory State (`@nan0web/db`).

2. **UI-Специфічні Snapshot-Тести:**
   Якщо пакет реалізує лише моделі — базового рівня достатньо. Але якщо в пакет додається шар UI (CLI, Web, Chat, Mobile), він зобов'язаний реалізувати **свої власні Snapshot-тести** для цих же самих сценаріїв (User Stories).
   - `CLI Adapter` -> `*.cli.story.js` (рендер ANSI / String снапшотів)
   - `Web Adapter` -> `*.web.story.js` (Playwright HTML/Image снапшоти)
   - `Mobile Adapter` -> `*.mobile.story.js`
   
Це гарантує, що базова бізнес-логіка генерує правильний візуальний стан у кожному середовищі. Людина лише мануально верифікує фінально зібрані снапшоти.

### Приклад: Повний сценарний тест для бізнес-додатку

```js
import { describe, it } from 'node:test'
import assert from 'node:assert/strict'
import { runGenerator, ask, progress, result } from '@nan0web/ui'
import DB from '@nan0web/db'
import { resolveDefaults } from '@nan0web/types'

// ─── 1. Доменна Модель (Model-as-Schema) ───

class TransferModel {
  static fromAccount = { help: 'Source account', default: '' }
  static toAccount = { help: 'Destination account', default: '' }
  static amount = {
    help: 'Transfer amount',
    default: 0,
    type: 'number',
    validate: (v) => (v > 0 ? true : TransferModel.UI.error_insufficient),
  }
  static UI = {
    error_insufficient: 'Insufficient funds',
  }

  constructor(data = {}) {
    Object.assign(this, resolveDefaults(TransferModel, data))
  }
}

// ─── 2. Бізнес-Флоу (Генератор) ───

class BankingFlowModel {
  static UI = {
    processing: 'Processing transfer...',
    success: 'Transfer completed',
    error_insufficient: 'Insufficient funds',
  }
  /** @type {DB} */
  db

  constructor({ db }) {
    this.db = db
  }

  async *run() {
    const balance = (await this.db.loadDocument('accounts/balance')) ?? {}
    const { value: transfer } = yield ask('transfer', TransferModel)

    const available = balance[transfer.fromAccount] || 0
    if (available < transfer.amount) {
      yield { type: 'log', level: 'error', message: BankingFlowModel.UI.error_insufficient }
      return result({ status: 'failed', reason: 'insufficient_funds' })
    }

    yield progress(BankingFlowModel.UI.processing)

    balance[transfer.fromAccount] = available - transfer.amount
    balance[transfer.toAccount] = (balance[transfer.toAccount] || 0) + transfer.amount
    await this.db.saveDocument('accounts/balance', balance)

    yield { type: 'log', level: 'success', message: BankingFlowModel.UI.success }
    return result({ status: 'ok', txId: `TX-${Date.now()}` })
  }
}

// ─── 3. Сценарні Тести ───

describe('BankingFlowModel OLMUI Scenario', () => {
  it('asks for transfer form as first intent', async () => {
    const db = new DB({ predefined: [] })
    await db.connect()
    const gen = new BankingFlowModel({ db }).run()
    const { value: intent } = await gen.next()
    assert.equal(intent.type, 'ask')
    assert.equal(intent.model, true)
    assert.equal(intent.schema, TransferModel)
  })

  it('rejects transfer with insufficient funds', async () => {
    const db = new DB({ predefined: [['accounts/balance', { UA001: 100 }]] })
    await db.connect()
    const events = []
    const data = await runGenerator(new BankingFlowModel({ db }).run(), {
      ask: async () => ({ value: { fromAccount: 'UA001', toAccount: 'UA002', amount: 999 } }),
      log: (i) => events.push(`${i.level}:${i.message}`),
      progress: () => {},
    })
    assert.equal(data.status, 'failed')
  })

  it('completes transfer and updates balances in DB', async () => {
    const db = new DB({
      predefined: [['accounts/balance', { UA001: 1000, UA002: 500 }]],
    })
    await db.connect()
    const data = await runGenerator(new BankingFlowModel({ db }).run(), {
      ask: async () => ({ value: { fromAccount: 'UA001', toAccount: 'UA002', amount: 300 } }),
      log: () => {},
      progress: () => {},
    })
    assert.equal(data.status, 'ok')
    const bal = await db.loadDocument('accounts/balance')
    assert.equal(bal.UA001, 700)
    assert.equal(bal.UA002, 800)
  })
})
```

1. **ЗАБОРОНЕНО** `fs.readFile`, `fs.writeFile`, `process.cwd()` — лише `@nan0web/db*`
2. **ОБОВЯЗКОВО** `new DB({ predefined: [...] })` для тестів — це In-Memory адаптер
3. **ОБОВЯЗКОВО** `db.connect()` перед будь-якими операціями
4. **ОБОВЯЗКОВО** перевіряти стан через `db.loadDocument()` після мутації
5. **ОБОВЯЗКОВО** UI-повідомлення — тільки з `static UI = {}` моделі (i18n-ready)
6. **ОБОВЯЗКОВО** `ask(fieldName, Model)` для складних форм (повертає instanceof Model)
7. **ЗАБОРОНЕНО** хардкодити тексти в генераторі — тільки `Model.UI.key`

   У парадигмі OLMUI **Модель = Цілий Додаток**. Тест моделі — це повноцінний прохід всіма її станами (перевірка `yield` інтенцій). 
2. **Що тестуємо (Екстенсивне покриття):** Тести мають перевіряти бізнес-логіку додатка, ніби ти "проклікуєш" інтерфейс як користувач, але за мілісекунди і без реального UI. Створи близько **30 сценаріїв**, які покривають кожну гілку генератора:

   - Замість рендерингу компонентів ти запускаєш метод моделі `model.run()` і симулюєш поведінку адаптера: зчитуєш кожен `yield` об'єкт, і відправляєш у `iterator.next({ value: '...' })` моковну відповідь.

# 📄 Шаблон: `README.md.js` — **ProvenDoc Manifest**

> 🧠 Це не просто тест.
> Це — **сценарій життя твого пакету**,
> записаний як приклади, покриті асертом.

## Ключові правила

1. Використовуй питання у тестах: `it("How to ...?")` — це потрібно для генерації datasets.
2. Використовуй `//import doSomething from "current-package"` у кожному блоці щоб цей приклад працював на 100%.
3. Використовуй `console.output()[0][1]` для перевірки значень де `output() => Array<Array<level: string, value: any>>`.
4. **Стоп-маркер DocsParser**: рядок що починається з `assert.` зупиняє збір документації.
   - **Бібліотечні пакети** (`co`, `db`, `i18n`): код між `*/` і `assert.` — це сам приклад. Нічого додаткового не потрібно.
   - **Серверні/API пакети** (`auth-node`, `http-node`): приклади в JSDoc як `curl` команди, а тестовий код (`fetch`) — реалізація що не повинна бути в README. В такому разі `assert.ok(api)` має бути **першим рядком** після `*/`.

## Структура файлу

````js

import assert from 'node:assert/strict'
import fsNode from 'node:fs'
import { fileURLToPath } from 'node:url'
import FS from '@nan0web/db-fs'
import { NoConsole } from '@nan0web/log'
import {
  DatasetParser, // .datasets з it("How to ...?")
  DocsParser, // .md з it("How to ...?")
  runSpawn, // для запуску команд
} from '@nan0web/test'
import {} from /* exports */ './index.js'

const fs = new FS()
let pkg

before(async () => {
  const doc = await fs.loadDocument('package.json', {})
  pkg = doc || {}
})

let console = new NoConsole()
beforeEach(() => {
  console = new NoConsole()
})

function testRender() {
  /**
   * @docs
   * # @nan0web/package-name
   * <!-- %PACKAGE_STATUS% -->
   * ## Description
   * ...
   * ## Installation
   */
  it('How to install with npm?', () => {
    /** ```bash
     * npm install @nan0web/package-name
     * ``` */
    assert.equal(pkg.name, '@nan0web/package-name')
  })

  /**
   * @docs
   * ## Usage
   * ### Basic Example
   */
  it('How to do something?', () => {
    //import { Something } from '@nan0web/package-name'
    const result = Something.from('test')
    console.info(result)
    assert.deepStrictEqual(console.output(), [['info', 'expected']])
  })

  /**
   * @docs
   * ## API
   * ### ClassName
   * * **Properties** ...
   * * **Methods** ...
   */
  it('All exported classes should pass basic test', () => {
    assert.ok(Something)
  })
}

describe('README.md testing', testRender)

describe('Rendering README.md', async () => {
  const parser = new DocsParser()
  const sourceCode = fsNode.readFileSync(fileURLToPath(import.meta.url), 'utf-8')
  const text = String(parser.decode(sourceCode))
  await fs.saveDocument('README.md', text)
  const dataset = DatasetParser.parse(text, pkg.name)
  await fs.saveDocument('.datasets/README.dataset.jsonl', dataset)

  it('document is rendered in README.md', async () => {
    const text = await fs.loadDocument('README.md')
    assert.ok(text.includes('## License'))
  })
})

## Bun-сумісність

- **ЗАБОРОНЕНО**: Передавати `Function` в `DocsParser.decode()` — Bun (JavaScriptCore) стрипає коментарі з `Function.toString()`.
- **ОБОВ'ЯЗКОВО**: Читати вихідний файл як рядок через `fs.loadDocument('src/README.md.js', '')` і передавати рядок у `parser.decode(source)`.

**`task.spec.js`** — контрактні тести:

- **ПРАВИЛЬНО**: `import { MyClass } from '../src/MyClass.js'` → `assert.equal(instance.method(), expected)`
- **НЕПРАВИЛЬНО**: `execSync('node --test src/MyClass.test.js')` — це делегування, а не контракт
- Кожен `it()` має **напряму імпортувати** модуль і тестувати конкретну поведінку
- `execSync` допустимий лише для тестування CLI output (`node bin/cli.js --help`)

| Повний       | все                                                | `npm run test:all`      | Pre-commit gate           |

## 3. CLI Sandbox — `play/main.js`

```javascript
#!/usr/bin/env node
import process from 'node:process'
import Logger from '@nan0web/log'
import { select } from '@nan0web/ui-cli'
import DBFS from '@nan0web/db-fs'

const dataFs = new DBFS({ root: 'play/data', cwd: import.meta.dirname + '/..' })
let translations = {}

async function loadLocale(locale) {
  const uri = locale === 'en' ? 'index' : `${locale}/index`
  const doc = await dataFs.fetch(uri)
  translations = doc?.t ?? {}
}

function t(key) {
  return translations[key] || key
}

// ─── Demo Scenarios ─────────────────────────
async function demoBasic() {
  /* ... */
}
async function demoAdvanced() {
  /* ... */
}

// ─── DEMO_MAP (обов'язковий для snapshot-тестів) ─────
/** @type {Record<string, () => Promise<void>>} */
const DEMO_MAP = {
  basic: demoBasic,
  advanced: demoAdvanced,
  all: async () => {
    for (const [k, fn] of Object.entries(DEMO_MAP)) {
      if (k !== 'all') await fn()
    }
  },
}

function getArg(flag) {
  const prefix = `--${flag}=`
  const arg = process.argv.find((a) => a.startsWith(prefix))
  return arg ? arg.slice(prefix.length) : undefined
}

async function main() {
  const argLang = getArg('lang')
  const argDemo = getArg('demo')
  await loadLocale(argLang || 'en')

  // Headless mode (snapshot tests)
  if (argDemo) {
    const fn = DEMO_MAP[argDemo]
    if (!fn) {
      console.error(`Unknown: ${argDemo}`)
      process.exit(1)
    }
    await fn()
    return
  }

  // Interactive mode — select menu loop
  const log = new Logger({ level: 'info', icons: true, chromo: true })
  while (true) {
    const options = [...Object.keys(DEMO_MAP).filter((k) => k !== 'all'), '← Exit']
    const choice = await select({ title: t('Select demo:'), options, console: log })
    if (options[choice.index].startsWith('←')) break
    await DEMO_MAP[Object.keys(DEMO_MAP)[choice.index]]()
  }
}

process.on('SIGINT', () => process.exit(0))
main().catch((err) => {
  console.error(err)
  process.exit(1)
})
```

**Ключове**: `DEMO_MAP` + `--demo=`/`--lang=` = headless mode для snapshot-тестів.

# ⚡ SSG Gallery Viewer (Static Iframe Architecture)

У рамках екосистеми OLMUI ми зберігаємо легкі текстові HTML-фрагменти (SSG-снепшоти) у папці `snapshots/ssg/`. Для того, щоб QA-інженери або розробники могли зручно "проклікати" та візуально оцінити ці зліпки, ми генеруємо швидкий локальний **SSG Gallery Viewer**.

## 🏗️ Проблематика (Media Queries та Ізоляція)

Збережені SSG-снепшоти — це "голі" HTML-блоки (без `<html>` та `<head>`). Якщо їх просто вставити у `div` (навіть з `width: 375px`), виникають три критичні проблеми:

1. **Поведінка Media Queries:** CSS-правила типу `@media (min-width: 768px)` (які керують сіткою Bootstrap чи Tailwind) спрацьовують відштовхуючись від реальної ширини *вікна браузера*, а не ширини `div`-контейнера. Молекула `375px` буде рендеритись як десктоп.
2. **Конфлікти CSS-фреймворків (Agnostic):** Пакет може використовувати Bootstrap, інший — Lit Web Components, третій — Tailwind. Ми не можемо просто підключити один файл стилів на всю сторінку галереї.
3. **Глобальний Scope:** Ідентифікатори зліпків можуть перетинатися з UI самого в'юера галереї.

## 🛠 Архітектурне Рішення: "Isolated iframe + Markdown Hierarchy"

Для того, щоб забезпечити **чесний** рендер HTML-фрагментів, використовується архітектура ізольованих iframe, які вбудовуються у Markdown або окремий HTML-рендер.

1. Галерея генерує ієрархію в `snapshots/ssg/` (аналогічно Web: `locale/theme/size/component`).
2. У **кожній** папці генерується файл `index.md` з навігацією `⬅ Назад`.
3. SSG-зліпки — це чисті HTML-файли (`.html` body). VSCode підтримує їх перегляд, але для точного `@media` рендеру галерея додатково створює `SSG_GALLERY.html` (Local Only) або інжектує `<iframe width="375" height="800"></iframe>` для чесного viewport.
4. Всередині iframe інжектується базова структура:
   ```html
   <!DOCTYPE html>
   <html lang="uk" data-bs-theme="dark">
   <head>
     <!-- Vite Native CSS Injection (Single Source of Truth) -->
     <script type="module" src="/src/ui/react-bootstrap/theme.js"></script>
   </head>
   <body>
     {SSG_HTML_CONTENT}
   </body>
   </html>
   ```

## 🎨 Zero-Logic Style Injection (theme.js)

> **ОБОВ'ЯЗКОВО** для всіх додатків, що використовують SSG Gallery Viewer.

### Проблема

Коли SSG-зліпок рендериться в iframe, йому потрібні повні стилі додатка (Bootstrap, кастомний CSS, шрифти). Існують три підходи:

| Підхід | Fidelity | HMR | Maintenance |
|--------|----------|-----|-------------|
| ❌ Хардкод CDN + `<link>` | Низька (розсинхронізація) | Ні | Постійна ручна синхронізація |
| ❌ Окрема CSS-збірка | Висока (після rebuild) | Ні | Додатковий конвеєр |
| ✅ **Vite Native (theme.js)** | **100% клон додатка** | **Так** | **Налаштував 1 раз** |

### Рішення: Style Entrypoint

Кожен додаток **обов'язково** виділяє всі CSS-імпорти в окремий файл `theme.js`:

```js
// src/ui/react-bootstrap/theme.js — ЄДИНЕ ДЖЕРЕЛО ПРАВДИ ДЛЯ СТИЛІВ
import 'bootstrap/dist/css/bootstrap.min.css'
import './styles.css'
```

Основний entry point (`main.jsx`) підключає стилі через нього:

```js
// main.jsx
import './theme.js'
import App from './App.jsx'
```

### Як це працює в SSG Gallery

Всередині кожного iframe в'юер підключає `<script type="module" src="/src/ui/react-bootstrap/theme.js">`.
Vite бачить цей запит, компілює всі CSS-залежності з `theme.js`, і **нативно інжектує стилі** у `<head>` iframe без запуску React, Router чи API.

**Результат:**
- ✅ **100% відповідність стилів** реальному додатку
- ✅ **Vite HMR:** зміна в `styles.css` миттєво оновлює всі iframe
- ✅ **Zero Logic Overhead:** жодного React, Store чи API-дзвінку

### ⚠️ Gotcha: Екранування `</script>` у srcdoc

Коли `srcDoc` формується всередині `<script>` блоку HTML-файлу, HTML-парсер бачить `</script>` і **передчасно закриває зовнішній скрипт**. Рішення:
```js
// ❌ Зламає парсер:
`<script src="theme.js"></script>`

// ✅ Безпечно:
`<script src="theme.js"><${'/'}script>`
```

# 🕵️ Subagent Protocol (Zero-Hallucination API)

Цей workflow визначає правила створення та взаємодії із вузькоспеціалізованими **Субагентами** (Subagents) всередині екосистеми NaN0Web. Головна мета — подолання "контекстної сліпоти" великих моделей шляхом мікророзподілу завдань, а також створення універсальних, платформонезалежних (Agnostic) AI API, які можна викликати з будь-якого середовища (Antigravity, Cursor, CI/CD, Node.js клієнт чи звичайний HTTP).

Згідно з резолюцією **Ради Мудреців** (фільтрація 4 законами логіки), субагент є суворо ізольованою, безстанційною функцією (stateless function), яка є "ультразвуковим дефектоскопом" для коду чи даних.

## 1. Архітектура Субагента (The Agnostic Blueprint)

Субагент не є "співрозмовником". Він — строгий механізм. Його контракт описується у вигляді Markdown або JSON інструкції, яка ідеально вміщується у `System Prompt`.

### Обов'язкові Вхідні Параметри (Input Payload):

- **Пайлоад має бути мінімалістичним і строго визначеним.** Жодних зайвих файлів або історії діалогів.
- Для валідації UI (`snapshot-audit`), пайлоад складається з:
  1. `[SPEC_LINK] / [RULES]`: Посилання на специфікацію або набір правил.
  2. `[LOCALE] / [CONTEXT]`: Мова або вузький контекст (`uk`, `en`, `theme`).
  3. `[TARGET_CONTENT]`: Сирий вміст файлу або зліпка (`.snap`, `.webp`, `.js`).

### Формат Спілкування (Persona & Execution):

Системний промпт субагента має бути агресивним на відсікання галюцинацій:

> "You are an isolated Zero-Hallucination UI Validator. You compute logical facts. You do not code. You do not talk. Your only objective is to compare `[TARGET]` against `[RULES]` in context of `[LOCALE]`.
> Any missing translations (e.g. English labels in `uk` locale), technical leakage (`NaN`, `[object Object]`, `undefined`), or structure mismatches are CRITICAL errors."

## 2. Суворі Обмеження Виводу (Output Contract)

Згідно з законом достатньої підстави та законом виключеного третього, субагент повертає лише строгий JSON, за яким машина може прийняти детерміноване рішення (Pass/Fail).

```json
{
  "score": 85,
  "errors": [
    "Line 12: Untranslated literal 'Create New Variant' strictly violates 'uk' locale rule.",
    "Line 15: Critical artifact '[object Object]' found inside rendering block."
  ]
}
```

- **`score` (0-100)**: Точна метрика відповідності. Тільки `100` означає успіх (Green).
- **`errors`**: Обов'язкове підкріплення помилок (Sufficient Reason) із конкретними номерами рядків або ключами.
- **Відсутність тексту**: Відповідь, що містить відформатований текст поза межами `{"score", "errors"}` (на кшталт "Here is your validation..."), вважається поламаним контрактом.
- **Кешування**: Результати успішних відповідей автоматично кешуються у `.agent/cache/subagents` на основі SHA-256 хешу промпту і вхідного файлу (`target`). Використовуйте `force: true` або `--force`, щоб обійти кеш. Кешування працює лише для субагентів, якщо `target` є шляхом до файлу, або пакетом файлів. Виклик через паралельні процеси використовує одну базу кешу, запобігаючи повторним дорогим запитом до LLM (зберігає ресурси і час як AI, так і розробника).

   - **Що робить:** Перевіряє TypeScript (запускає `tsc` перевірку для гарантії, що всі типи та JSDoc вірні).
   - **Коли використовувати:** Як тільки ти змінив чи оновив типи JSDoc (`/** @param ... */`), або змінив аргументи, що впливають на типізацію.

4. **`npm run test:e2e`**
   - **Що робить:** Повільні Playwright тести в браузері.
   - **Коли використовувати:** Тільки коли змінено реальний DOM, інтерактивність UI компонентів чи CSS. Логіка додатку перевіряється в `test:unit`, а тут лише *візуальний/браузерний рендеринг*. Запускається ЯК ОКРЕМИЙ ТЕСТ тільки після успіху `test:unit`.

## 🛡️ Стандартні комбіновані скрипти (Commit / Release)

Ці скрипти об'єднують перевірки на різних етапах готовності:

1. **`npm test` (або `npm run test`)**
   - Це стандартна базова перевірка, яка визначає чи пройде **git commit** (через husky hook).
   - Включає в себе: `test:unit`, `test:ssg` та `build`. 
   - Використовуй тоді, коли хочеш перевірити, чи все готово до наступного коміту.

2. **`npm run test:all`**
   - Об'єднує **всі абсолютно** тести, включаючи повільні: `test`, `test:docs`, `test:play`, `npm run test:e2e` та `npm run knip`.
   - Запускається **виключно перед генеральним релізом (`/npm-release`) або публікацією**, або в кінці масштабного завдання. Під час ітеративної розробки не використовуй `test:all`, оскільки він займає багато часу!

---

# UI-CLI Component Standards

All console inputs and outputs in CLI applications, agents, or CLI generators (like `generate.js`) MUST use the official `@nan0web/ui-cli` components.

## Rules

1. **Output Information**
   - NEVER use raw `console.log` for structure or tables.
   - USE: `Alert`, `Table`, `ProgressBar`, `Spinner`, and other view components available in `@nan0web/ui-cli`.

2. **Input Information**
   - NEVER use raw `readline` directly if `ui-cli` is accessible.
   - USE: `Input`, `Select`, `renderForm` and other interactive prompt components from `@nan0web/ui-cli`.

3. **Application to Other UI Frameworks**
   - This strict schema-driven input/output interface pattern also extends to other environments.
   - `ui-lit`, `ui-chat`, `ui-swift`, and others should implement an equivalent logical `Model as Schema` (e.g., `AppConfig` class) directly into standardized visual UI components.

## Example (Model as Schema inside AppConfig)

```javascript
import { createT } from '@nan0web/i18n'
import { select } from '@nan0web/ui-cli'

// словник має завантажуватись з data/*/t.yaml, наприклад:
// const doc = await db.fetch('uk/index', {})
const t = createT(doc.t)

export class AppConfig {
  static overwrite = { help: t('Overwrite (replace everything)'), default: false }
  static merge = { help: t('Merge (add missing files only)'), default: false }
}

async function askAction() {
  const res = await select({
    label: t('Directory already exists. What should we do?'),
    options: [
      { label: t(AppConfig.overwrite.help), value: 'overwrite' },
      { label: t(AppConfig.merge.help), value: 'merge' },
    ],
  })
  return res.value
}
```

Коли агент має намір використати інструменти для запису або читання файлів поза межами поточного робочого простору (workspace) або коли потребує підтвердження/допомоги від клієнта, він **ПОВИНЕН** завжди ініціювати голосове повідомлення за допомогою інструменту `run_command` з флагом `SafeToAutoRun: true`.

⚠️ **НАЙГОЛОВНІШЕ ПРАВИЛО:** Повідомлення має бути **дуже коротким** (1-2 коротких речення). Жодних довгих абзаців. Користувач сам прочитає конкретні деталі на екрані, йому потрібно лише прийти на звук.

# 🕸 Web Gallery (Playwright Auditor)

У рамках екосистеми OLMUI (One Logic — Many UI) ми обов'язково повинні генерувати візуальну галерею `WEB_GALLERY.md`, яка містить скріншоти (WebP зображення) React/Lit UI-компонентів у браузері. Вона використовується для мануального QA та візуальної верифікації.

## 🏗️ Структура галереї

Галерея має сувору ієрархію, що генерується безпосередньо в директорії `snapshots/`. Це дозволяє зручно переглядати всі снапшоти через вбудований Markdown-рев'ювер VSCode, уникаючи засмічення папки `docs/`.

Структура для Web виглядає так:
`snapshots/web/{locale}/{theme}/{size}/{component_or_flow}/`

У **кожній** папці цієї ієрархії генерується файл `index.md`, який має обов'язкове посилання `⬅ Назад` на рівень вище (прямо на початку документа).

1. **Кореневий рівень `snapshots/web/index.md`**:
   - Посилання на локалізовані галереї: `[uk](./uk/index.md)`, `[en](./en/index.md)`.
   - Зведена таблиця аудиту компонентів та помилок.

2. **Мовний індекс `snapshots/web/{locale}/index.md`**:
   - Навігація вгору: `[⬅ Назад](../index.md)`.
   - Список тем: `☀️ light`, `🌙 dark`, `⚡ contrast`.

3. **Тема індекс `snapshots/web/{locale}/{theme}/index.md`**:
   - Навігація вгору: `[⬅ Назад](../index.md)`.
   - Список ширин (viewports): `📱 375`, `💊 768`, `💻 1024`, `🖥 1200`, `📺 1920`.

4. **Розмір індекс `snapshots/web/{locale}/{theme}/{size}/index.md`**:
   - Навігація вгору: `[⬅ Назад](../index.md)`.
   - Список компонентів або User Stories (`component_or_flow`).

5. **Рівень компонента `snapshots/web/{locale}/{theme}/{size}/{component_or_flow}/index.md`**:
   - Навігація вгору: `[⬅ Назад](../index.md)`.
   - Вбудовані WebP скріншоти (напр., `![Step 1](./step-1.webp)`).

> ⚠️ **ВАЖЛИВО:** Всі згенеровані картинки `*.webp` повинні бути додані до `.gitignore`, щоб не роздувати репозиторій. У Git зберігаються лише `index.md` файли (по суті діють як текстова база відрендереного дерева). Інші формати (наприклад `ssg`) можуть зберігати лише `HTML body` замість картинок.

## 🔍 Аудит-перевірки (5 рівнів)

Тест-аудитор виконує 5 автоматичних перевірок для кожного компонента:

1. **Block Exists** — Чи існує DOM-блок з `id="block-{component}"`. Штраф: −10.
2. **Preview Area** — Чи присутній `.example-preview` контейнер. Штраф: −3.
3. **Zero-Hardcode Text** — Чи немає сирих `[T]` mock-перекладів у кнопках. Штраф: −15.
4. **Undefined/Null/NaN** — Чи немає витоку `undefined`, `null`, `[object Object]`, `NaN` у видимих текстах. Штраф: −20.
5. **Locale Mismatch** — Чи немає сирих i18n-ключів (`.dot.notation`) або англійського тексту в `uk`-локалі. Штраф: −15.

## 🏗️ Як додати новий компонент

1. Створити файл пісочниці `src/ui/{adapter}/components/{Component}.play.jsx` (або `.play.js` для Lit).
2. Додати назву компонента (з `.play`) до масиву `COMPONENTS` у `e2e/web-gallery.test.js`:
   ```javascript
   const COMPONENTS = [
     'CardProduct.play',
     'CardDetails.play',
     'NewComponent.play', // ← додати сюди
   ]
   ```
3. Запустити `npm run test:web-gallery` для генерації скріншотів.
4. Перевірити `WEB_GALLERY.md` — новий компонент має з'явитись у таблиці аудиту.

## 🛡️ Правила верифікації

- **Zero Raw Keys:** У скріншотах галереї не має бути сирих i18n-ключів (наприклад, `cards.order.submit`). Всі ключі повинні бути перекладені.
- **Ізоляція:** Кожен компонент рендериться в ізольованому режимі (`?isolate=block-{name}`) без повного додатку (без Header/Footer/Sidebar).
- **Детермінізм:** Скріншоти знімаються після `waitForLoadState('networkidle')` та додаткового буферу, щоб уникнути flaky-результатів через асинхронне завантаження i18n.
- **Формат:** Скріншоти зберігаються у форматі WebP (або PNG як fallback) для мінімального розміру.
