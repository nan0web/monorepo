# 🏗️ Проєкт: @nan0web/auth.app

> **Призначення:** Універсальне ядро автентифікації та контролю доступу (Identity & Access Management) для Sovereign екосистеми NaN•Web. Реалізація принципу "One Logic — Many UI" для керування користувачами, ролями та правилами доступу.

---

## 🧭 Фаза 1: Філософія та Абстракція (The Seed)

### 1. Місія та Філософія

**Місія:** Забезпечити суверенний контроль над ідентичністю та доступом, де право на авторизацію належить власнику вузла, а не централізованому провайдеру.

**Філософія:**
- **Identity Integrity:** Ідентичність користувача — це набір валідованих властивостей (email, soulId), а не просто рядок у БД.
- **Polymorphic Access:** Доступ має перевірятися однаково в CLI, Web та API через єдиний доменний шар.
- **Fail-Fast Security:** Якщо інфраструктура (DB, Logger) не завалена ("Interface Welding"), додаток не запуститься.

### 2. Абстракція (Base App)
- **Core:** `AuthApp` — ядро логіки, що наслідує `Model`. Використовує асинхронні генератори (`yield*`) для потокової передачі статусів.
- **UI Adapters:**
    - `AuthCLI` — інтерактивне керування через термінал.
    - `AuthRouter` — автоматичне підключення до HTTP-серверів на основі доменних повідомлень.
    - `AuthWeb` — (майбутнє) Lit/React компоненти для кастомних форм.

## 📐 Фаза 2: Доменне Моделювання (v1.4.0)

### 1. Моделі (Model-as-Schema)

#### 🛡 AuthPolicy (extends Model)
Визначає правила доступу до ресурсів за допомогою glob-патернів.
- `protectedPaths` — патерни захисту (напр. `/api/**`).
- `publicPaths` — виключення (напр. `/api/health`).
- **Метод `isProtected(path)`**: перевірка за Законом Тотожності.

#### 👤 UserAccount (extends Model)
Доменна сутність користувача.
- `verified` — підтверджено емейл / DID.
- `approved` — схвалено адміністратором спільноти.
- `soulId` — суверенний міст ідентичності.

#### ⚙️ AuthConfig (extends Model)
Налаштування поведінки системи.
- `verificationFlow` — стратегія реєстрації (`email-only`, `admin-only`, `email+admin`).
- `passwordMinLength` — правила валідації.

### 2. Повідомлення (Intent Messages)
Кожна дія — це доменне повідомлення (`SignUpMessage`, `LoginMessage`), яке валідує себе перед відправкою в логіку.

## 🛠 Фаза 3: Верифікація Логіки (Dispatcher)

**Статус:** ✅ Завершено (v1.4.0)

Реалізовано **Polymorphic Dispatcher `run(msg)`**:
- Додаток автоматично маршрутизує вхідні повідомлення на відповідні методи логіки на основі їхнього імені.
- Потік виводу (`OutputMessage`) забезпечує зворотний зв'язок у реальному часі ("Registration successful" ⇢ "Check email").

## 🪐 Фаза 4: Onboarding Strategies

### 1. Стратегії Реєстрації
- **Sovereign (`email-only`)**: Швидка реєстрація з авто-затвердженням після коду в пошті.
- **High-Trust (`email+admin`)**: Підтвердження пошти + ручне схвалення модератором.
- **Private (`admin-only`)**: Створення акаунта тільки адміністратором, або реєстрація без коду з очікуванням схвалення.

### 2. Interface Welding (Fail-Fast)
Контракт інфраструктури перевіряється в конструкторі. Відсутність `db` або `tokenManager` призводить до негайної зупинки.
