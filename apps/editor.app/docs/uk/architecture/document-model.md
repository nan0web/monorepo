# 📄 Уніфікована Модель Документа (Document-as-Truth)

> Будь-який контент — це документ у базі даних з єдиною структурою, посиланнями (`$ref`) та підтримкою злиття між додатками.

---

## Визначення

Document-as-Truth означає, що **немає** окремих моделей для різних типів контенту. Замість цього існує єдиний клас `Document`, який через конфігурацію та успадкування адаптується до будь-якого випадку використання.

## Структура документа

```js
class Document extends Model {
  static $url          // Унікальний URL документа
  static $deleted      // 0 = активний, 1 = видалений
  static $hidden       // 0 = видимий, 1 = прихований
  static title         // Заголовок
  static description   // Опис
  static content       // Структурований об'єкт (теги, значення, графи)
  static image         // URL зображення
  static icon          // Іконка
  static metrics       // Довільні метрики (object)
  static order         // Вага сортування (число, за замовчуванням 0)
}
```

## Посилання між документами (`$ref`)

Документи пов'язуються між собою через механізм `$ref`. В архітектурі NaN•Web посилання вказуються **без розширення файлу**:

```yaml
# data/articles/architecture.nan0
title: Архітектура NaN•Web
author:
  $ref: authors/yaro
related:
  - $ref: articles/polymorphism
  - $ref: articles/validation
tags:
  - $ref: tags/architecture
```

### Як працює резолвінг посилань:
1. Система отримує шлях `authors/yaro`.
2. DB-адаптер пробує знайти файл, додаючи розширення по черзі: `.nan0`, `.md`, `.yaml`, `.jsonl`.
3. Перший знайдений файл завантажується та перетворюється на об'єкт.
4. Якщо вказано **Hash (#)**, система витягує конкретне поле:
   - `$ref: authors/yaro#name` → повертає рядок "Ярослав Снігірьов".
   - `$ref: project/stats#metrics.latency` → повертає вкладене значення "0 ms".
5. Це дозволяє створювати гранулярні зв'язки між документами, зберігаючи дані суверенними.

## Суверенність даних та додатки

Дані є **первинними** та суверенними. Структура папок має бути зрозумілою навіть без додатків, які їх обробляють.

### Проблема "Path-based Routing"
Раніше ми припускали, що додаток визначається за шляхом (напр. `data/editor/`). Це обмежує суверенність. 

### Новий підхід: Alias та Контекст
Додатки об'єднуються у спільний простір. Якщо виникає конфлікт імен моделей, використовуються аліаси.

```yaml
# nan0web.nan0
apps:
  editor:
    package: @nan0web/editor.app
    alias: editor
  blog:
    package: @nan0web/blog.app
    alias: blog
```

Контекст (яка модель обробляє дані) визначається не жорстким шляхом, а типом контенту та налаштуваннями у `_.nan0`.

```yaml
# data/shared/_.nan0
# Дані тут суверенні, але ми вказуємо, який "інструмент" 
# найкраще підходить для їх обробки за замовчуванням.
preferred_app: editor
```

## Зміст документа (Content)

Поле `content` — це не просто текст. Це **граф об'єктів**:
- У форматі `.nan0` це вкладені структури.
- У форматі `.md` це Markdown, який парситься в об'єктне представлення.
- Це дозволяє редактору маніпулювати контентом як деревом даних, а не просто як рядком.

## Аналоги

| Платформа | Модель контенту | Уніфікація |
| :--- | :--- | :--- |
| **WordPress** | Post, Page, Custom Post Type | Часткова |
| **Strapi** | Content Types (JSON Schema) | Висока |
| **Notion** | Block-based | Повна (все — блок) |
| **Salesforce** | Standard + Custom Objects | Висока |
| **NaN•Web** | Document + `$ref` + Alias | Повна (все — документ) |
