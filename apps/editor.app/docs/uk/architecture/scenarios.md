# 🎬 Канонічні Сценарії (Canonical Scenarios)

Цей документ визначає поведінку `@nan0web/editor.app` через набір сценаріїв, які мають бути покриті тестами доменної моделі.

---

## 1. Управління документами та Посилання ($ref)

### Сценарій 1.1: Створення документа з посиланнями
- **Дія**: Користувач створює новий документ `articles/new-post` та додає автора через `$ref: authors/yaro`.
- **Очікуваний результат**: 
    - Документ успішно створюється у `stageDb`.
    - Метод `resolveReferences()` повертає повний об'єкт автора.
    - Валідація проходить успішно.

### Сценарій 1.2: Видалення документа з активними посиланнями (Referential Integrity)
- **Дія**: Спроба видалити `authors/yaro`, на якого посилається 10 статей.
- **Очікуваний результат**: 
    - Система повертає `ModelError` з попередженням про наявність 10 посилань.
    - Видалення блокується, якщо в конфігурації вказано `strictReferences: true`.

---

## 2. Успадкування та Конфігурація (_.nan0)

### Сценарій 2.1: Каскадне злиття налаштувань
- **Дія**: Запит налаштувань для `data/apps/editor/doc-1`.
- **Очікуваний результат**:
    - Отримано об'єкт, де глобальні параметри з `data/_.nan0` злиті з локальними з `data/apps/editor/_.nan0`.
    - Локальні параметри мають пріоритет.

---

## 3. Стейджинг та Конфлікти

### Сценарій 3.1: Поле-рівневий стейджинг
- **Дія**: Зміна лише поля `title` у документі, де вже є змінений `content` у `mainDb`.
- **Очікуваний результат**:
    - У `stageDb` зберігається лише дельта (diff) для поля `title`.
    - Візуальний маркер (●) відображається лише біля заголовка.

### Сценарій 3.2: Паралельне редагування (Conflict Resolution)
- **Дія**: Два "користувачі" (або процеси) одночасно змінюють один документ.
- **Очікуваний результат**:
    - При спробі коміту система виявляє невідповідність `baseHash`.
    - Пропонується інтерфейс розв'язання конфліктів (Merge View).

---

## 4. Пошук та Масштабування

### Сценарій 4.1: Пошук серед 1 млн документів
- **Дія**: Виконання запиту `find('tag:architecture')` у базі з 1,000,000+ записів.
- **Очікуваний результат**:
    - Час відповіді < 100мс (використання In-memory індексів).
    - Використання пам'яті не перевищує ліміт (шардинг індексів).

---

## 5. UI Адаптери та Інтенти

### Сценарій 5.1: Послідовний запит (CLI Flow)
- **Дія**: Модель генерує інтент `ask('title')`.
- **Очікуваний результат**:
    - CLI адаптер виводить запит.
    - Після вводу невірних даних (пуста стрічка) модель повертає `ModelError`.
    - CLI адаптер повторює запит саме для цього поля.

### Сценарій 5.2: Паралельний рендер (Web Flow)
- **Дія**: Модель генерує набір інтентів для всієї форми.
- **Очікуваний результат**:
    - Web адаптер рендерить всі поля одночасно.
    - Помилки валідації відображаються паралельно біля кожного поля.
