# Anti-Gravity Release Protocol

// turbo-all

Цей workflow перетворює технічне завдання користувача на реліз з контрактними тестами.
Усі команди безпечні для автономного запуску (git commit/push залишаються заблокованими через RULES.md).

## Кроки

### 1. Дослідження контексту (Mandatory)

// turbo

Перед створенням релізу ОБОВ'ЯЗКОВО перевір екосистему:

1. Прочитай `packages/index.md` та `apps/index.md` щоб уникнути дублювання.
2. Використовуй `grep` або `find` для пошуку існуючих рішень.

Визнач версію та scope. Перевір існуючі релізи (шлях: `releases/{major}/{minor}/vX.Y.Z/`):
// turbo

```
find releases/ -name "*.spec.js" -o -name "*.test.js" 2>/dev/null || echo "Немає релізів"
```

Якщо користувач не вказав версію — визнач її автоматично (наступний minor від останньої).

### 2. Створення контракту (Contract-First)

На основі ТЗ користувача створи два файли в `releases/{major}/{minor}/vX.Y.Z/`:

**`task.md`** — місія релізу:

- Що саме робимо (scope)
- Acceptance criteria (Definition of Done)
- **Architecture Audit (Чекліст)**:
  - [ ] Чи прочитано Індекси екосистеми?
  - [ ] Чи існують аналоги в пакетах?
  - [ ] Джерела даних: YAML, nano, md, json, csv?
  - [ ] Чи відповідає UI-стандарту (Deep Linking)?

**`task.spec.js`** — контрактні тести:

- КОЖЕН пункт ТЗ = один або більше `it()` тестів
- Тести ПОВИННІ ПАДАТИ (Red) перед початком реалізації
- Джерела правди: дані з `data/` у форматах `.yaml, .nano, .md, .json, .csv`
- Використовуй `node:test`, `assert/strict`

⚠️ **ВАЖЛИВО: `task.spec.js` — це ПРЯМІ функціональні тести, а не мета-тести.**

- **ПРАВИЛЬНО**: `import { MyClass } from '../src/MyClass.js'` → `assert.equal(instance.method(), expected)`
- **НЕПРАВИЛЬНО**: `execSync('node --test src/MyClass.test.js')` — це делегування, а не контракт
- Кожен `it()` має **напряму імпортувати** модуль і тестувати конкретну поведінку
- `execSync` допустимий лише для тестування CLI output (`node bin/cli.js --help`)

**Архітектура тестів:**

| Шар          | Файли                  | Скрипт                   | Роль                      |
| :----------- | :--------------------- | :----------------------- | :------------------------ |
| Unit тести   | `src/**/*.test.js`     | `npm test`               | TDD біля коду             |
| CLI E2E      | `bin/*.test.js`        | `npm run test:snapshot`  | Snapshot CLI output       |
| Release spec | `release/**/*.spec.js` | `npm run release:spec`   | Контракт на фічі релізу   |
| Регресія     | всі `*.spec.js`        | `npm run release:verify` | Закриті релізи = регресія |
| Повний       | все                    | `npm run test:all`       | Pre-commit gate           |

Після закриття релізу його `task.spec.js` стає **регресійним тестом** — він запускається через `release:verify` при кожному коміті.

Запусти spec щоб переконатись що КОЖЕН тест Red:
// turbo

```
npm run release:spec
```

**ЗУПИНИСЬ**: Покажи користувачу контракт та запитай підтвердження перед реалізацією.

### 3. Послідовна реалізація (Red → Green)

Для кожної задачі з ТЗ:

1. Знайди відповідний `it()` тест у spec
2. Модифікуй `src/` код — НІКОЛИ не змінюй `.spec.js`
   - Якщо неможливо вирішити задачу без зміни spec — **повідом як про помилку**
   - Позначай такі тести як `.skip` з коментарем чому
   - Залиш для подальшого розгляду Архітектором
3. Запусти spec після кожної задачі:
   // turbo

```
npm run release:spec
```

4. Запусти регресію закритих контрактів:
   // turbo

```
npm run release:verify 2>&1 || echo "Немає закритих контрактів"
```

5. Перейди до наступної задачі

Повторюй поки всі тести не стануть зеленими (або .skip).

### 4. Фінальна верифікація

Коли всі spec зелені, запусти повну перевірку:
// turbo

```
npm run release:spec
```

### 5. Повна перевірка (test:all)

Запусти `test:all` для підтвердження що ВСЕ працює — не лише контракт, а й існуючі тести та білд:
// turbo

```
npm run test:all 2>&1 || echo "test:all FAIL — виправляй!"
```

### 5.1 Перевірка .npmignore

Переконайся що `.npmignore` існує і не пропускає зайве у публікацію:
// turbo

```
npm pack --dry-run 2>&1 | tail -5
```

**Перевір**: файли `chat/`, `.datasets/`, `*.test.js`, `play/`, `releases/`, `system.md` НЕ мають бути в tarball.
Якщо `.npmignore` відсутній — створи за шаблоном з `system.md` секції "Гігієна Пакету".

### 6. Звіт Архітектору

**ЗУПИНИСЬ**. Повідом користувача:

- "✅ Всі тести зелені. Ready for Architect Review."
- Покажи підсумковий звіт (які тести, скільки pass/fail/skip)
- Якщо є `.skip` тести — чітко вкажи їх та чому
- НЕ оголошуй реліз "Stable" — це право тільки Архітектора

**Правило закриття релізу (Міграція у Регресію)**

Після схвалення релізу користувачем, його контрактні тести перетворюються на живі регресійні тести.
Оскільки релізів з часом стає багато, ми **зберігаємо 3-рівневу структуру** (major/minor) у новій директорії регресії:

1. Створи папку для версії: `mkdir -p src/test/releases/{major}/{minor}/vX.Y.Z`
2. Перемісти всі `*.spec.js` (щоб у `releases/` історично залишились лише `.md` та статика):
   ```bash
   mv releases/{major}/{minor}/vX.Y.Z/*.spec.js src/test/releases/{major}/{minor}/vX.Y.Z/
   ```
3. Перейменуй їх на `.test.js`, щоб вони автоматично підхопились через `npm run test:js`:
   ```bash
   for f in src/test/releases/{major}/{minor}/vX.Y.Z/*.spec.js; do mv "$f" "${f%.spec.js}.test.js"; done
   ```

**⚠️ Стратегія оптимізації регресії:**
Тести релізів (тепер регресія) накопичуються. Тому:

- Вони мають бути **максимально швидкими** (використовуй `import` і `assert`, уникай `execSync` де можливо).
- З часом (наприклад, при мажорних оновленнях) Архітектор може проводити рефакторинг теки `src/test/releases/`: видаляти застарілі модульні контракти, що дублюються, або об'єднувати їх.

Користувач (або агент) закриває реліз командою:

```
npm run release:close -- vX.Y.Z
```
